# 3.1 — Publish ZIPs to Maven Central + split the release workflow

Follow-up to [3.1-no-fat-jar.md](3.1-no-fat-jar.md).

## Motivation

Two problems with the post-3.0 single `do-release.yml`:

1. **The cross-platform ZIPs are not durably published.** They only travel as
   1-day GitHub artifacts between jobs. Pre-3.0 the `jsignpdf-distribution`
   module published its ZIP to Maven Central
   (`com/github/kwart/jsign/jsignpdf-distribution/2.3.0/jsignpdf-distribution-2.3.0.zip`),
   giving a permanent, verifiable, mirror-able download. We want that back for
   the full + minimal ZIPs.

2. **The release cannot be resumed.** `package`, `flatpak`, and
   `publish-release` all `needs: do-release` and consume its in-memory GitHub
   artifacts. If a platform runner pool starves and the run hits the 24h job
   ceiling / `timeout-minutes`, the only way forward is to redo the
   *irreversible* half — Maven `release:prepare`/`perform`, the version bump
   commits, the git tag, the Central deploy. That is unacceptable.

## Design

### Maven Central publication

`distribution` (packaging `pom`) attaches the two ZIPs as classified
artifacts via `build-helper-maven-plugin:attach-artifact`, phase `package`
(after the assembly executions, before `maven-gpg-plugin` at `verify`, so they
are GPG-signed and deployed by the existing `release` profile / `nexus-staging`
with `autoReleaseAfterClose=true`):

```
com.github.kwart.jsign:jsignpdf-distribution:<version>:zip:full
com.github.kwart.jsign:jsignpdf-distribution:<version>:zip:minimal
```

The assembly `<attach>false</attach>` flags stay; build-helper does the
attaching explicitly so the classifiers are controlled. GitHub Release /
SourceForge asset filenames are unchanged (`jsignpdf-<version>-full.zip` etc.);
only the Maven coordinates carry the `jsignpdf-distribution` artifactId.

### Workflow split

**`do-release.yml`** — the irreversible half, unchanged in spirit:
release-notes check → Java/GPG setup → `mvn clean install` →
`release:prepare` → `release:perform` (now also deploys the two ZIPs to
Central). All `upload-artifact` steps and the `package`/`flatpak`/
`publish-release` jobs are removed. On success a final job invokes
`package-release.yml` (`workflow_call`) passing `release-version`.

**`package-release.yml`** — new, idempotent, independently re-runnable:

- Triggers: `workflow_dispatch` (input `release-version`) **and**
  `workflow_call` (same input).
- `fetch-zips` job: `mvn dependency:copy` for the `full` and `minimal`
  classifiers, wrapped in a **bounded retry loop** to ride out Central
  propagation delay (artifact not instantly downloadable from repo1 after
  `release:perform`). Clean failure if it never appears. Re-publishes the ZIPs
  + extracted PDF guide as in-run artifacts.
- `package` matrix (windows-2022 / ubuntu-24.04 x2 / macos-14): checkout the
  release tag, download the `full` ZIP, unzip it, point
  the platform script at the unpacked `lib/` via `JSIGNPDF_LIB_DIR`. No local
  `mvn -pl distribution -am package` — the **exact published bits** are
  packaged.
- `flatpak` matrix: as before, sourcing the full ZIP from the download instead
  of a `do-release` artifact.
- `publish-release`: SHA-256 + SourceForge mirror + GitHub Release. PDF from
  inside the full ZIP (`docs/JSignPdf.pdf` → `JSignPdf-<v>.pdf`); README from
  the tag's `distribution/doc/release-notes/<base>.md`.

Re-dispatching `package-release.yml` with the same version re-downloads from
Central, rebuilds all installers, and recreates the GitHub Release (existing
`gh release delete` then `create` makes it idempotent).

### Platform script change

`build-{linux,macos,windows}-installers` get one change: the lib input
(`APPASSEMBLY` / `$appassembly`, default `distribution/target/appassembler/lib`)
is overridable by the `JSIGNPDF_LIB_DIR` env var. `package-release.yml` unzips
the Central full ZIP and sets `JSIGNPDF_LIB_DIR=<unzip>/jsignpdf-<v>/lib`. The
scripts already skip `javafx-*` jars (JFX comes from the bundled Zulu+FX
runtime), and `lib/*.jar` does not recurse into `lib/javafx/`, so the unpacked
ZIP yields the same non-JFX jar set as the appassembler staging dir. The PDF
guide is copied by the workflow into `distribution/target/pdf-guide/` (existing
path), so no guide-related script change is needed. Local
`./distribution/.../build-*-installers` runs are unaffected (env var unset →
default path).

### Intel-macOS dropped from the matrix

`macos-x64` is no longer built. Apple discontinued Intel Macs in 2023,
GitHub's last Intel hosted image (`macos-13`) is being retired, and an
attempted self-hosted Intel runner ran headlong into the actions-runner's
Node 24 requiring macOS ≥ 13.5 on hardware that can't reach it. An earlier
revision wired a `pick-intel-runner` probe job + dynamic `runs-on` to prefer
a self-hosted Intel mac with hosted fallback — that machinery is gone too.
Intel Mac users use the Apple Silicon DMG under Rosetta 2 or the
cross-platform full ZIP with their own Java 21. Only the Apple Silicon
(`macos-14`, `macos-aarch64` label) entry remains.

## Out of scope

- Changing what the ZIPs contain (covered by 3.1-no-fat-jar).
- macOS signing/notarization (still deferred).

## Decisions

- **Packaging input = exact Central ZIP**, not a tag rebuild — guarantees the
  installers wrap the published artifact.
- **Auto-chain + standalone rerun** — `do-release` calls `package-release` on
  success; `package-release` is also `workflow_dispatch`-able for recovery.
- **Poll Central until downloadable** — bounded retry in `fetch-zips`.
- **Drop `macos-x64` from the matrix** — Apple discontinued Intel; hosted
  pool retiring; the self-hosted path is not viable for retiring hardware.
- **Classifiers `full`/`minimal` on `jsignpdf-distribution`** — closest analog
  to the pre-3.0 single-ZIP coordinate.
