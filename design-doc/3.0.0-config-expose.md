# Expose advanced configuration in the JavaFX UI

Status: proposal
Target release: 3.0.0
Tracking: no dedicated issue (user-request driven; companion to the Presets feature)
Scope: JavaFX GUI + storage refactor; CLI inherits the storage refactor but gains no new flags

## 1. Context and motivation

JSignPdf has long had a second tier of configuration that lives next to the binaries:

- `<install>/conf/conf.properties` — global tweaks (font path, certificate-validation toggles, SSL relax flag, PDF preview library order, default TSA hash algorithm, `pkcs11config.path`).
- `<install>/conf/pkcs11.cfg` — sample SunPKCS11 provider configuration; pointed to by `pkcs11config.path`.

These were "edit the file, restart the app" knobs; the GUI never exposed them. With the XDG-style config directory (`<cfg>/`) introduced by the [Presets feature](3.0.0-presets-feature.md), there is now a natural place to store *user-owned* advanced configuration, and a Preferences dialog can finally surface those knobs in the UI.

This document is the merged plan from a Java backend pass and a JavaFX UI pass. It covers storage (where the data lives, how it loads, how migration works) and UX (where the dialog lives, what it looks like, what each control binds to).

## 2. Goals and non-goals

**Goals**

- Move the install-dir `conf.properties` configuration into `<cfg>/advanced.properties`, owned per-user.
- Move `pkcs11.cfg` to a fixed well-known location `<cfg>/pkcs11.cfg`. Drop the `pkcs11config.path` key entirely.
- Ship sane defaults inside `JSignPdf.jar` as resources so a fresh install needs no config file at all.
- Add a modal **Preferences** dialog accessible from the menu bar, with inline help text mirroring the comments in today's `conf.properties`.
- Apply changes immediately where possible; clearly label the few cases that require a restart.
- Stop shipping the `<install>/conf/` directory entirely. The bundled jar resources are now the single source of defaults; users edit only `<cfg>/`.

**Non-goals**

- No CLI flag changes. CLI continues to honor every setting through the same `AdvancedConfig` plumbing, but no new options are added.
- No layered "system + user" config (admin-pushed defaults). Open question §11 captures this; deferred.
- No JSON / YAML migration; format stays plain `java.util.Properties`.
- No editor for `pkcs11.cfg` beyond a raw multi-line text area. We surface the file as-is and let users follow Oracle's PKCS#11 reference.

## 3. Decisions settled

Pinned by the project owner before this doc was written:

| Decision | Choice |
|---|---|
| UI surface | Single modal **Preferences** dialog opened from the menu bar. |
| PKCS#11 editor | Plain multi-line monospace TextArea, no validation. |
| Static-final config reads | Refactor to read dynamically; apply on Save where feasible. |
| Migration | None for advanced config. The `<install>/conf/` directory is no longer shipped; users edit `<cfg>/` only. |
| Storage filename | Different from `config.properties`. Recommendation pinned in §4: **`advanced.properties`**. |
| `pkcs11config.path` key | Removed. Path is fixed at `<cfg>/pkcs11.cfg`. |
| Defaults | Bundled jar resources are the in-jvm fallback and the source for "Reset to defaults". |

## 4. File layout and key inventory

After migration:

```
<cfg>/
  config.properties          # main signer options (Presets feature)
  advanced.properties        # NEW — user-owned advanced configuration (optional)
  pkcs11.cfg                 # NEW — well-known PKCS#11 provider config (optional)
  presets/
    preset-*.properties
```

**Filename = `advanced.properties`.** Justification: `config.properties` is taken (main signing options); `app.properties` is too generic. "Advanced" matches the user-facing label of the Preferences dialog.

### 4.1 Bundled jar resources

Path | Contents
---|---
`/net/sf/jsignpdf/conf/advanced.default.properties` | Shipped defaults for every key in §4.2. In-jvm fallback when a key is missing from the user file. Source for "Reset to defaults".
`/net/sf/jsignpdf/conf/pkcs11.cfg.sample` | Sample SunPKCS11 provider configuration. Used as the "Reset to bundled sample" source in the Preferences PKCS#11 tab.

Both ship under `jsignpdf/src/main/resources/net/sf/jsignpdf/conf/`. Maven already includes `src/main/resources`; no `pom.xml` change.

### 4.2 advanced.properties keys

`pkcs11config.path` is **removed**. The remaining keys, with the live defaults bundled in `advanced.default.properties`:

| Key | Type | Default | Description |
|---|---|---|---|
| `font.path` | String (path) | *(empty — bundled DejaVuSans used)* | Path to TTF/OTF font for visible-signature L2 text. |
| `font.name` | String | basename of `font.path` | Font name passed to OpenPDF `BaseFont.createFont`. |
| `font.encoding` | String | *(empty — `WINANSI` if path set, `Identity-H` for bundled)* | `Cp1250`, `Cp1252`, `Cp1257`, `Identity-H`, `Identity-V`, `MacRoman`. |
| `certificate.checkValidity` | boolean | `true` | If true, expired / not-yet-valid certs are filtered from the alias list. |
| `certificate.checkKeyUsage` | boolean | `true` | If true, only certs with `digitalSignature`/`nonRepudiation` KU are listed. |
| `certificate.checkCriticalExtensions` | boolean | `false` | If true, certs with critical extensions outside JSignPdf's known set are rejected. |
| `relax.ssl.security` | boolean | `false` | If true, disables hostname verification and trusts all server certs for TSA/OCSP/CRL. |
| `pdf2image.libraries` | String (CSV) | `jpedal,pdfbox,openpdf` | Ordered preview backends. First success wins. Allowed tokens: `jpedal`, `pdfbox`, `openpdf`. |
| `tsa.hashAlgorithm` | String | `SHA-256` | Default TSA digest when not overridden in signer options. |

The bundled defaults are the **live 3.0.0** defaults — i.e. what `Constants.java` already returns when the user file is absent. The pre-3.0.0 install-dir `conf/conf.properties` (with `relax.ssl.security=true`, `tsa.hashAlgorithm=SHA-1`, `pdf2image.libraries=jpedal,pdfbox,pdfrenderer`) is no longer shipped or migrated — see §8.

## 5. Storage backend

### 5.1 New class: `AdvancedConfig`

`net.sf.jsignpdf.utils.AdvancedConfig`. Replaces the existing `ConfigProvider` singleton.

```java
public final class AdvancedConfig {
    AdvancedConfig(Path userFile, Properties bundledDefaults);
    public synchronized void   reload();           // re-read userFile; defaults stay cached
    public synchronized void   resetToDefaults();  // delete userFile, drop user state
    public synchronized Set<String> save() throws ProperyProviderException;
    // ↑ returns the keys that changed since the last load/save — used by the change-bus

    // overlay getters: userFile wins, bundled default fills in
    public String  getProperty(String key);
    public String  getProperty(String key, String def);
    public String  getNotEmptyProperty(String key, String def);
    public boolean getAsBool(String key, boolean def);
    public int     getAsInt(String key, int def);
    public float   getAsFloat(String key, float def);

    // mutators (used by Preferences)
    public void setProperty(String key, String value);
    public void setProperty(String key, boolean value);
    public void removeProperty(String key);  // revert one key to bundled default
}
```

Internally, two `PropertyProvider` instances:
- one bound to `<cfg>/advanced.properties` (mutable, persists on `save()`),
- one bound to `null` and seeded from `/net/sf/jsignpdf/conf/advanced.default.properties` (read-only).

Every getter consults user → defaults in that order. The existing `PropertyProvider` typed-getter API is preserved; only the class name on call sites changes.

### 5.2 Factory wiring

Add to the existing `PropertyStoreFactory` (introduced in the Presets feature):

```java
public AdvancedConfig advancedConfig();   // cached, lazy-init
```

Implementation: resolves `<cfg>/advanced.properties` via `ConfigLocationResolver`, loads bundled defaults from `getResourceAsStream` once, hands both to `AdvancedConfig`. The factory's existing `resetAll()` clears this cache too.

The Presets feature explicitly broke `PropertyProvider.getInstance()`. We follow the same shape: `AdvancedConfig` is *cached*, not statically singleton — tests can construct an isolated `new AdvancedConfig(tmpPath, defaults)` directly.

### 5.3 `AppConfig` static accessors

`net.sf.jsignpdf.utils.AppConfig` is a thin static facade so call sites stay compact:

```java
public final class AppConfig {
    public static boolean relaxSslSecurity()        { return cfg().getAsBool("relax.ssl.security", false); }
    public static String  pdf2imageLibraries()      { return cfg().getNotEmptyProperty("pdf2image.libraries", PDF2IMAGE_LIBRARIES_DEFAULT); }
    public static String  defaultTsaHashAlg()       { return cfg().getNotEmptyProperty("tsa.hashAlgorithm", "SHA-256"); }
    public static boolean checkValidity()           { return cfg().getAsBool("certificate.checkValidity", true); }
    public static boolean checkKeyUsage()           { return cfg().getAsBool("certificate.checkKeyUsage", true); }
    public static boolean checkCriticalExtensions() { return cfg().getAsBool("certificate.checkCriticalExtensions", false); }
    public static String  fontPath()                { return cfg().getNotEmptyProperty("font.path", null); }
    public static String  fontName()                { return cfg().getNotEmptyProperty("font.name", null); }
    public static String  fontEncoding()            { return cfg().getNotEmptyProperty("font.encoding", null); }
    private static AdvancedConfig cfg() { return PropertyStoreFactory.getInstance().advancedConfig(); }
}
```

The string constants `PDF2IMAGE_LIBRARIES_DEFAULT`, `PDF2IMAGE_JPEDAL`, `PDF2IMAGE_PDFBOX`, `PDF2IMAGE_OPENPDF` stay in `Constants` (literals, no config dependency).

### 6.1 Delete from `Constants`

| File | Today | After |
|---|---|---|
| `ssl/SSLInitializer.java` | `Constants.RELAX_SSL_SECURITY` | `AppConfig.relaxSslSecurity()` |

`ConfigProvider` is deleted entirely. `IOUtils.findFile` no longer reads anything under `<install>/conf/`. The remaining caller is `PKCS11Utils.registerProviders(String)` for user-supplied paths.

### 6.3 Hot-readable vs. restart-required

After the refactor, **most settings take effect immediately** because their consumers re-read on every use:

- `relax.ssl.security` (in `DynamicX509TrustManager.checkServerTrusted`)
- `pdf2image.libraries` (in `Pdf2Image`)
- `tsa.hashAlgorithm`
- `certificate.check*`
- `font.*` — *with* the small `FontUtils.reset()` hook described below.

**Restart-required** cases are limited to two:

1. **PKCS#11 provider registration.** `Signer.main` calls `PKCS11Utils.registerProviders(...)` once at startup. Re-registering at runtime is supported but `PKCS11Utils.unregisterProviders()` warns about driver hangs and sleeps 1s. We do not hot-reload. The Preferences dialog labels the PKCS#11 tab with "Takes effect after restart".
2. **`SSLInitializer.init()`** runs once at startup and sets system properties (`jsse.enableSNIExtension`, `sun.security.ssl.allowUnsafeRenegotiation`, …) plus a default `HostnameVerifier`. Toggling `relax.ssl.security` post-init affects the live trust-decision branch but cannot retract the system properties. We surface a "Takes effect after restart" hint on the relax-SSL toggle and accept that the hot path is partially applied.

### 6.4 `FontUtils.reset()` hook

`FontUtils.l2baseFont` is a static cache populated on first use. Add:

```java
public static synchronized void reset() {
    l2baseFont = null;
}
```

`AdvancedConfig.save()` returns the set of changed keys (§5.1). The Preferences-dialog code inspects that set; if any `font.*` key changed, it calls `FontUtils.reset()`. The next `getL2BaseFont()` rebuilds from the new path/name/encoding lazily. Result: font changes apply on next preview / next sign without a restart hint.

## 7. PKCS#11 wiring and well-known location

### 7.1 New helper

```java
// in PKCS11Utils
public static void registerProvidersFromDefaultLocation() {
    Path cfg = ConfigLocationResolver.getInstance().getConfigDir();
    if (cfg == null) return;
    Path file = cfg.resolve("pkcs11.cfg");
    if (Files.isRegularFile(file)) {
        registerProviders(file.toString());
    }
    // else: skip silently — same as today's empty-path branch
}
```

Add a `getPkcs11ConfigFile()` accessor on `ConfigLocationResolver` next to `getMainConfigFile()`, and a constant `PKCS11_CONFIG_FILE_NAME = "pkcs11.cfg"`.

`Signer.main` (line 105) becomes:
```java
PKCS11Utils.registerProvidersFromDefaultLocation();
```

No FX-only wiring needed; FX startup runs through `Signer.main` → `FxLauncher.launch`.

### 7.2 Behavior on missing/empty/unparseable file

- **Missing**: skip silently. Matches today's `StringUtils.isEmpty(configPath)` early-return.
- **Empty**: `SunPKCS11` constructor throws; `PKCS11Utils.initPkcs11Provider` already catches and logs SEVERE. Same as today.
- **Unparseable**: same as empty — caught, logged, no provider registered, app continues.

## 8. Migration

Driven from `ConfigLocationResolver.resolveAndMigrate()` (existing first-launch hook). It copies `~/.JSignPdf` → `<cfg>/config.properties` only when the target dir doesn't exist:

```
ConfigLocationResolver.resolveAndMigrate():
  1. Resolve target <cfg>; bail if user.home unset.
  2. If <cfg> exists → done, return it.
  3. Files.createDirectories(<cfg>, <cfg>/presets).
  4. If ~/.JSignPdf exists → copy → <cfg>/config.properties.
```

**No automatic migration of the pre-3.0.0 install-dir `conf/` directory.** Starting with 3.0.0 the distribution no longer ships `<install>/conf/`; users with a hand-edited legacy `conf.properties` or `pkcs11.cfg` from an earlier release are expected to re-enter their tweaks via the Preferences dialog (or copy the file by hand to `<cfg>/`). This is called out in the 3.0.0 release notes.

Rationale: the migration heuristic from earlier drafts (byte-equality of the legacy install-dir file against a bundled sample) was unreliable across version bumps — a 3.0.0 build whose bundled defaults differ from any earlier release would mass-copy unedited legacy files into `<cfg>/advanced.properties`, pinning out-of-date values for every upgrading user. Cutting the install-dir read entirely is the smaller blast radius.

## 9. UI design — Preferences dialog

### 9.1 Menu placement

**Preferences…** lives in the **File** menu, between **Reset Settings** and **Exit**:

```
File
  Open PDF…
  Close
  ──
  Save Output As…
  ──
  Recent Files ▶
  ──
  Reset Settings…
  Preferences…   (Shortcut+,)
  ──
  Exit
```

Justification: a single Preferences item does not justify a dedicated top-level menu. The File menu is the conventional home for application-level settings in single-window apps; placing it just above Exit keeps it reachable without cluttering the menu bar.

Accelerator `Shortcut+Comma` maps to `⌘,` on macOS (native convention) and `Ctrl+,` elsewhere. No existing JSignPdf binding uses `Comma`.

i18n: `jfx.gui.menu.file.preferences` = "Preferences…", `jfx.gui.preferences.title` = "Preferences".

### 9.2 Dialog skeleton

- `PreferencesDialog extends Dialog<Boolean>` (Boolean result = "anything saved", caller decides whether to refresh derived UI).
- `initOwner(stage)`, `initModality(APPLICATION_MODAL)`, `setResizable(true)`.
- Default size 720×560, min 640×480.
- Buttons: **Reset section to defaults** (left, `OTHER`), **Cancel** (`CANCEL_CLOSE`), **OK** (`OK_DONE`). No Apply — six small sections, no preview, the modal-state cost outweighs the benefit.
- ESC and window-close = Cancel.
- Stylesheet: reuse `jsignpdf.css`.

**Layout = `TabPane` with 6 tabs.** Six independent sections; most users visit one. The PKCS#11 textarea is large (~20 rows) and would dominate a single scrolling form; isolating it in a tab keeps other sections compact. A left-side master/detail is overkill for a fixed 6-section set.

```
+---------------------------------------------------------------+
| Preferences                                              [X]  |
+---------------------------------------------------------------+
| [ Visible Signature | Certificates | Network | PDF Render |   |
|   TSA | PKCS#11 ]                                             |
+---------------------------------------------------------------+
|                                                               |
|  Font file:    [ /usr/share/fonts/...    ] [Browse…]          |
|     Path to a TTF/OTF font used to render the visible         |
|     signature text. Leave empty to use the bundled default.   |
|                                                               |
|  Font name:    [ vladimir.ttf            ]                    |
|     Font name as known by the file above.                     |
|                                                               |
|  Encoding:     [ Identity-H       v ]                         |
|     Cp1250, Cp1252, Cp1257, Identity-H, Identity-V, MacRoman. |
|                                                               |
+---------------------------------------------------------------+
|  [Reset section to defaults]               [Cancel]   [ OK ]  |
+---------------------------------------------------------------+
```

FXML root: `Preferences.fxml` (`net/sf/jsignpdf/fx/view/`), controller `PreferencesController` in `net.sf.jsignpdf.fx.preferences`. Buttons added via `getDialogPane().getButtonTypes()`, mirroring `ManagePresetsDialog`.

### 9.3 Help-text rendering

**Inline helper text under each field**, CSS class `pref-help` (small, muted). Tooltips would hide explanatory prose behind a hover; a description column would force a 2-column grid. Inline helper text is the convention in macOS/GNOME Preferences and makes the dialog self-documenting.

Long lines: `Label wrapText="true" maxWidth="Infinity"`. Helper bundle values are pre-shortened to ~120–180 chars distilled from the legacy `conf.properties` comments — long-form prose is no longer shipped (the install-dir file was retired in 3.0.0).

The "Takes effect after restart" hint is a small Label (CSS `pref-restart-hint`) with an info icon, placed inline at the right of the affected control's row. i18n: `jfx.gui.preferences.restartHint` and `.restartHint.tooltip`.

#### 9.4.6 PKCS#11 — `tabPkcs11`

**Restart hint applies.**

Layout:

1. `Label fx:id="lblPkcs11Path"` — wrapped, monospace, shows `Editing: <cfg>/pkcs11.cfg` (absolute path resolved at open-time). i18n template `jfx.gui.preferences.pkcs11.pathLabel` = `Editing: {0}`.
2. Restart-hint label.
3. `TextArea fx:id="txtPkcs11Body"` — `wrapText=false`, `prefRowCount=20`, `prefColumnCount=80`, `VBox.vgrow=ALWAYS`, monospace via CSS `pref-monospace`. Both scrollbars visible by default.
4. `Label` with `jfx.gui.preferences.pkcs11.help`.
5. Button row: `Button fx:id="btnPkcs11ResetSample"` text `%jfx.gui.preferences.pkcs11.resetSample` — replaces textarea content with the bundled `/net/sf/jsignpdf/conf/pkcs11.cfg.sample`. No confirmation prompt; Cancel still discards.

Empty-content semantics: if the user clears the textarea, OK still saves an empty file. A label below the help text — bound to `txtPkcs11Body.textProperty().isEmpty()` — reads `jfx.gui.preferences.pkcs11.emptyHint` = "Empty: PKCS#11 provider will not be registered."

The bundled sample is exposed to the UI via `PKCS11Utils.getSampleConfig()` (returns the resource as `String`).

### 9.5 ViewModel and controller

- `PreferencesViewModel` (POJO with JavaFX properties): `StringProperty fontPath, fontName, fontEncoding, tsaHashAlgorithm, pkcs11Body;` `BooleanProperty checkValidity, checkKeyUsage, checkCriticalExtensions, relaxSslSecurity;` `BooleanProperty pdfLibJpedal, pdfLibPdfbox, pdfLibOpenpdf;` plus three `IntegerProperty` ordering fields (1/2/3) used to derive the comma-separated string on save.
- `PreferencesViewModel(AdvancedConfig snapshot, String pkcs11Body)` populates from the snapshot.
- `Snapshot toSnapshot()` builds the write-back payload for `AdvancedConfig.setProperty(...)` + the new pkcs11 body.
- `PreferencesController.show(Stage owner)` is the static entry point: loads FXML, builds the VM from `PropertyStoreFactory.getInstance().advancedConfig()` + `Files.readString(cfg.resolve("pkcs11.cfg"))` (or empty), calls `controller.bind(vm)`, shows the dialog. On OK: validate; on success, write the keys back via `AdvancedConfig.setProperty` (the change-set returned by `save()` drives `FontUtils.reset()`); also write the pkcs11 body to `<cfg>/pkcs11.cfg` via `Files.writeString`.

**Dirty tracking: do not gate OK on dirty.** Always-enabled OK is simpler, more predictable for users, and lets "Reset to defaults → OK" do the obvious thing. `AdvancedConfig.save()` is a cheap diff; a no-op save is fine.

Keyboard: tab order = FXML order; ESC = Cancel; Enter triggers OK *unless* focus is on `txtPkcs11Body` (TextArea inserts newline — desired).

## 11. Testing strategy

### Unit

`AdvancedConfigTest`:
- Bundled-default fallback when user file is absent.
- User-file overrides for every key.
- `reload()` picks up out-of-band edits.
- `resetToDefaults()` deletes user file and reverts getters.
- `removeProperty(k)` reverts a single key to bundled default.
- `save()` returns the correct change-set.

`AppConfigTest`: smoke test of every accessor against a stubbed `AdvancedConfig`.

`AdvancedMigrationTest` (extends existing `ConfigLocationResolverTest`):
- Fresh install with no legacy files anywhere → `<cfg>` created, presets dir created, `advanced.properties` and `pkcs11.cfg` are NOT created.
- Accessor surface: `getAdvancedConfigFile()` and `getPkcs11ConfigFile()` resolve to the expected paths under `<cfg>`.
- Bundled jar resources `advanced.default.properties` and `pkcs11.cfg.sample` are present and contain expected sentinel keys.

`SignerPkcs11Test`:
- `<cfg>/pkcs11.cfg` absent → `registerProvidersFromDefaultLocation()` is a no-op, no exception.
- Syntactically invalid `pkcs11.cfg` → logs SEVERE, does not throw.

`FontUtilsResetTest`:
- `getL2BaseFont()` populates cache; `reset()` nulls it; next call rebuilds.

`PreferencesViewModelTest`:
- Snapshot → VM round-trip across all properties.
- `applyDefaults()` loads jar resource defaults onto every property.
- PDF library order encoding: default → `jpedal,pdfbox,openpdf`; one disabled → `jpedal,pdfbox`; reordered → respects new order.
- `pkcs11Body` round-trips byte-for-byte (whitespace not auto-trimmed).
- Validation pure functions (`PreferencesValidation.validateFontPath`, `validatePdfLibSelection`).

JavaFX-thread UI bindings are not covered by tests (repo does not use TestFX). Stay at the VM/service level.

### Integration / manual

- Open dialog with fresh config: every field shows the bundled default; no errors.
- Edit each section → OK → reopen reflects on disk and in UI.
- Cancel after editing → reopen unchanged.
- Per-section "Reset section to defaults" restores defaults for active tab only; other tabs preserved.
- Font path: invalid path → per-field error; valid path → saves; visible-signature preview rebuilds with the new font without restart.
- PDF render: uncheck all → `pdfNoLibrary` validation; reorder → preview honors new order on next render.
- TSA dropdown change persists across restarts.
- Network: toggle SSL relax → restart-hint badge visible.
- PKCS#11: load bundled sample, edit, OK, reopen — bytes match. Clear textarea fully → empty-hint label visible; saving an empty file works; on next launch no PKCS#11 provider is registered.
- Menu accelerator `Shortcut+,` opens the dialog.
- Drag-and-drop into the main window does not bypass modality.
- macOS sanity: accelerator displays as `⌘,`, no menu-bar bleed.

## 12. Open questions and future work

1. **Boolean polarity for `certificate.check*`.** Today's `KeyStoreUtils` reads `getAsBool(key, true)` — i.e. the keys are *positive* (`=true` enables, `=false` disables). The Preferences UI labels them positively too. Confirm we're not silently flipping semantics on existing users who have `certificate.checkValidity=false` in their install-dir file: after migration, the value `false` flows into `<cfg>/advanced.properties` unchanged, and the CheckBox renders unchecked. ✅ no change.
2. **`AppConfig` package.** Placed in `net.sf.jsignpdf.utils` for symmetry with `PropertyStoreFactory`. An argument exists for the top-level package since these are app-wide knobs. No strong preference; flag for review.
3. **CLI scope.** After the refactor the CLI also reads `<cfg>/advanced.properties` (no longer the install-dir file). That matches the "install dir is template, user data lives in `<cfg>/`" model from the Presets feature. Confirm intentional.
4. **Layered "system + user" config.** Out of scope. If admin-pushed defaults become a requirement, the layering would be: bundled jar defaults → install-dir `conf.properties` (admin) → `<cfg>/advanced.properties` (user). The current design merges admin into user via the migration heuristic; subsequent admin changes are not picked up. Defer.
5. **Hot reload of `relax.ssl.security`.** Decision §6.3 is "do not attempt SSL re-init; show restart hint". Alternative would be `SSLInitializer.reinit()` swapping the default `SSLSocketFactory`, but the system-property side effects cannot be retracted in-process. Confirm we accept the restart hint.
6. **Migration heuristic.** Bytes-equal vs. SHA-256. Bytes-equal is simpler and adequate for these tiny files. Confirm we don't need property-by-property comparison for `conf.properties`.
7. **Apply button.** The dialog has OK/Cancel/Reset — no Apply. Six small sections, no preview surface. Confirm.
