# Read passwords from standard input (stdin) in CLI mode

Status: proposal
Target release: 3.0.0
Tracking: [GitHub issue #7](https://github.com/intoolswetrust/jsignpdf/issues/7)
Scope: CLI only — GUI unchanged

## 1. Context and motivation

Today every password used by the JSignPdf CLI must be supplied as an option value on the command line:

| Short      | Long                   | Meaning                                     |
|------------|------------------------|---------------------------------------------|
| `-ksp`     | `--keystore-password`  | Keystore password                           |
| `-kp`      | `--key-password`       | Private-key password inside the keystore    |
| `-opwd`    | `--owner-password`     | PDF owner password (when encryption is on)  |
| `-upwd`    | `--user-password`      | PDF user password (when encryption is on)   |
| `-tscp`    | `--tsa-cert-password`  | TSA client PKCS#12 password                 |
| `-tsp`     | `--tsa-password`       | TSA HTTP-auth password                      |

Passing a secret as a cmdline argument leaks it to every local process that can read `/proc/<pid>/cmdline` (or to anyone scrolling shell history). The original feature request (issue #7) is to let a daemon feed the keystore password to JSignPdf over stdin instead of on the command line.

`java.io.Console.readPassword` was considered by the reporter and rejected — it is `null` when the JVM's stdin is not a TTY (as is the case when JSignPdf is spawned by another process), so it does not solve the daemon scenario. We still want to use `Console` for the *interactive* case because it disables echo.

This document specifies a single, explicit mechanism that works for both a piped daemon and an interactive human, covers all six password fields, and is straightforward to test.

## 2. Design goals and non-goals

**Goals**

- Allow any of the six password options above to be supplied via stdin, individually or in combination.
- Make ordering unambiguous and discoverable (printed at runtime, documented up front).
- Preserve full backwards compatibility: without the new flag, current behavior (including a literal `-` as a password value) is unchanged.
- Work identically in pipe mode (no TTY) and in TTY mode, except that TTY mode disables input echo and prints a prompt.
- Be testable without forking a process or touching the real `System.in` / `System.console()`.

**Non-goals**

- No changes to the GUI code path.
- No new `env:` / `file:` indirections. Those are valid alternatives (see §10) but are deliberately out of scope for this change.
- No change to the set of existing flags and their short/long names.

## 3. User-facing specification

### 3.1 New opt-in flag

A new CLI switch:

| Long                         | Short | Argument | Default |
|------------------------------|-------|----------|---------|
| `--enable-stdin-passwords`   | *(none)* | *(no arg, boolean)* | off |

When the flag is **not** present, the parser behaves exactly as today: every password option value is taken verbatim from the command line, and a literal `-` is a valid (if unusual) password.

When the flag **is** present, the string `-` (exactly one hyphen, nothing else) used as the value of any of the six password options becomes a **sentinel** meaning "read this password from stdin now". Any other value — including `-abc`, `abc-`, `—` (em-dash), or empty — is treated as a literal password exactly as before. Note: Apache Commons CLI intercepts a bare `--` as the end-of-options marker before we see the value, so `--` cannot be used as a literal password via `-ksp --` — that is a limitation of the parser, not of this feature.

### 3.2 Canonical read order

When more than one password option uses `-`, values are consumed from stdin in the following fixed order, regardless of their order on the command line:

1. `-ksp`  / `--keystore-password`
2. `-kp`   / `--key-password`
3. `-opwd` / `--owner-password`
4. `-upwd` / `--user-password`
5. `-tscp` / `--tsa-cert-password`
6. `-tsp`  / `--tsa-password`

This is the natural order in which `SignerOptionsFromCmdLine.loadCmdLine()` already processes the options and matches the typical signing workflow (keystore → key → encryption → TSA).

Rationale for a fixed order instead of CLI-appearance order: scripts that conditionally include some options (e.g. only add `-tsp` when a TSA is configured) would otherwise have to recompute stdin line order on every change. A fixed order lets callers always feed passwords in the same sequence and simply skip lines that are not used.

### 3.3 Runtime feedback

Before reading each password the implementation prints one line to **stderr** of the form:

```
[jsignpdf] Reading password for --keystore-password (1/3) from stdin...
```

Where `(1/3)` is the 1-based index among password fields that were marked with `-` in this invocation, so the operator can verify the order matches what their pipe is feeding. The message is printed **before** blocking on the read, in both TTY and non-TTY mode. It uses stderr so it does not interfere with anything the caller is doing on stdout. The prefix `[jsignpdf]` makes it greppable.

When `--quiet` (`-q`) is set the line is suppressed — consistent with how `ARG_QUIET` already silences the rest of the logger.

### 3.4 TTY (interactive) behavior

If `System.console()` returns non-null, the input is treated as interactive:

- A prompt `Enter password for --keystore-password: ` is written to the console.
- The password is read via `Console.readPassword(...)`, which disables echo.
- The progress line from §3.3 is still emitted to stderr for consistency.

If `System.console()` is null (typical daemon case), the input is treated as piped:

- One line is read from `System.in` using a `BufferedReader` on `System.in` with the platform default charset.
- The newline terminator is stripped; the rest of the line is the password.
- An EOF or I/O error while reading a required password aborts the run with a clear error message (see §3.6).

### 3.5 Interaction with properties files

`ARG_LOADPROPS` (`-lp`) / `ARG_LOADPROPS_FILE` (`-lpf`) are processed *before* CLI options (see `SignerOptionsFromCmdLine.loadCmdLine()` around lines 95–106). A properties file can therefore set a password to `-`, but that `-` is *not* reinterpreted by this feature — the properties loader is not on the CLI parsing path. The stdin sentinel is strictly a CLI-option behavior.

If a password is set by a properties file and then the same option is given on the CLI with value `-`, the CLI wins (current behavior is unchanged — the CLI branch always overwrites the props value), and the CLI read-from-stdin path triggers.

### 3.6 Error and warning handling

A literal `-` as a password value is valid today and remains valid when `--enable-stdin-passwords` is not set (backwards compatibility, §3.1). Because that combination is almost always a mistake in practice — the user meant to read from stdin but forgot the flag — the parser emits a **warning** on stderr so the operator notices, but does *not* fail:

| Case                                                                              | Behavior                                                                                                                                     |
|-----------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------|
| A password option value equals `-` but `--enable-stdin-passwords` was not given   | Warn once per affected option, use `-` as the literal password value. Message: `[jsignpdf] Warning: --keystore-password value is '-'. Did you mean to pass --enable-stdin-passwords to read it from stdin? Using '-' as the literal password.` |
| Stdin EOF before the expected password line was read                              | Fail with non-zero exit: `Unexpected end of input while reading password for --keystore-password`                                            |
| IOException while reading                                                         | Fail with non-zero exit: `Failed to read password for --keystore-password: <cause>`                                                          |

The warning is suppressed by `--quiet` (`-q`), consistent with the progress line in §3.3.

## 5. Implementation plan

### 5.1 New class `net.sf.jsignpdf.StdinPasswordReader`

Small, self-contained, testable:

```java
final class StdinPasswordReader {
    static final String STDIN_SENTINEL = "-";

    StdinPasswordReader(BufferedReader reader, Console console, PrintStream progressOut, boolean quiet);

    /** Reads the next password; logs progress to progressOut. */
    char[] readNext(String optionLongName, int index, int total) throws IOException;
}
```

- In production it is instantiated with `new BufferedReader(new InputStreamReader(System.in))`, `System.console()`, and `System.err`.
- In tests it is instantiated with a `StringReader`, a null console, a `ByteArrayOutputStream`-backed `PrintStream`, and the desired quiet flag.
- No static dependency on `System.in` / `System.err` inside the class — all pulled from the constructor.

### 5.2 Changes in `SignerOptionsFromCmdLine`

Add a new constant pair in `Constants`:

```java
public static final String ARG_ENABLE_STDIN_PWDS_LONG = "enable-stdin-passwords";
```

(no short alias; long-only is fine for an opt-in safety switch.)

Register the flag in the static `OPTS` initializer at the top of `SignerOptionsFromCmdLine`:

```java
OPTS.addOption(OptionBuilder.withLongOpt(ARG_ENABLE_STDIN_PWDS_LONG)
    .withDescription(RES.get("hlp.enableStdinPasswords"))
    .create());
```

Then refactor the six existing `if (line.hasOption(ARG_…_PWD)) setXxxPasswd(line.getOptionValue(ARG_…_PWD));` blocks into a two-pass flow inside `loadCmdLine()`:

1. **Pass 1 — collect.** Walk the six options in canonical order. For each, record a tuple `(longName, rawValue, setter)` where `rawValue` is `line.getOptionValue(ARG)` or `null` if not present.
2. **Pass 2 — resolve.**
   - If `--enable-stdin-passwords` was given: compute `total` = count of tuples whose `rawValue` equals `-`, and for each such tuple call `reader.readNext(longName, index, total)` and pass the result to the setter. All other tuples pass their raw value through unchanged.
   - If the flag was **not** given: every `-` value is treated as a literal password (backwards compatible). For each such tuple, emit the one-shot warning from §3.6 and then pass `"-"` to the setter.

The refactor keeps the existing `setXxxPasswd(String)` signatures; the stdin-read path calls them with a `new String(char[])` to match (the `char[]` returned from `Console.readPassword` is converted only at this boundary).

Touched locations (existing line numbers in `SignerOptionsFromCmdLine.java`):

- Lines 130–131 (ksp), 136–137 (kp), 163–166 (opwd/upwd), 218–219 (tscp), 223–224 (tsp) — replaced by the unified pass above.
- Static initializer block starting around line 278 — adds the new `addOption(...)` call.

No other call sites need to change: `BasicSignerOptions.setKsPasswd(String)` etc. already exist and are used by the GUI too.

### 5.3 Security hygiene

- The new class accepts passwords via `char[]` internally, but since the downstream setters take `String` (current API), there is no net improvement in memory-zeroization; we will not pretend otherwise, and we will not change public setter signatures in this change. Improving keystore-password handling end-to-end is a separate, larger cleanup.
- The progress line on stderr never contains the password value.
- The interactive prompt uses `Console.readPassword` so terminal echo is off.
- No password is ever written to the standard logger.

## 6. Testing strategy

### 6.1 Unit tests — `StdinPasswordReaderTest`

In `jsignpdf/src/test/java/net/sf/jsignpdf/StdinPasswordReaderTest.java`:

- Reads a single password from a `StringReader("secret\n")`.
- Reads three passwords in order from `StringReader("a\nb\nc\n")`.
- Handles CRLF (`"a\r\nb\r\n"`).
- Handles a final line without a trailing newline.
- EOF before read → throws with a message that names the option.
- Progress line is written to the injected `PrintStream` and contains the long option name and the `(i/n)` marker.
- `quiet=true` suppresses the progress line.

### 6.2 Parser integration tests — extend `SignerOptionsFromCmdLineTest` (create if missing)

Drive `SignerOptionsFromCmdLine` end-to-end by:

- Injecting a canned `InputStream` in place of `System.in` for the duration of the test (via a seam on `SignerOptionsFromCmdLine` — e.g. a package-private constructor or static hook that takes a `StdinPasswordReader`), and asserting:
  - `-ksp -` with the new flag → keystore password equals the first stdin line.
  - `-ksp - -tsp -` with the new flag → ksp gets line 1, tsp gets line 2, regardless of CLI order.
  - `-ksp - -tsp - -kp -` → three lines consumed in canonical order (ksp, kp, tsp).
  - `-ksp -` *without* the new flag → keystore password is the literal `"-"` (backwards compatible) **and** a warning that names both the option and `--enable-stdin-passwords` is written to the captured stderr.
  - `-ksp -` without the new flag combined with `-q` → literal `"-"` is used, warning is suppressed.
  - `-ksp secret` with the new flag → `secret` is used verbatim; no stdin read happens.
  - `-ksp -abc` with the new flag → literal `-abc` is used; no stdin read (only exactly `-` is the sentinel).
- Asserting that a mix of properties-file and CLI passwords still honors CLI precedence and that a `-` coming from a properties file is *not* treated as a sentinel.

### 6.3 Manual verification before release

Out of scope for the JUnit suite but captured in the PR checklist:

- `echo "p" | java -jar jsignpdf.jar --enable-stdin-passwords -ksp - … input.pdf` signs successfully.
- Running in an interactive terminal shows the prompt and does not echo the password.
- Running on Windows `cmd.exe` and PowerShell behaves the same (line-ending handling).

## 7. Examples for the README

**Piped daemon, one password:**

```sh
printf '%s\n' "$KS_PASSWORD" \
  | jsignpdf --enable-stdin-passwords \
      -ksf keystore.p12 -ksp - -ka mykey \
      -d out/ input.pdf
```

**Piped daemon, keystore + TSA auth:**

```sh
{ printf '%s\n' "$KS_PASSWORD"; printf '%s\n' "$TSA_PASSWORD"; } \
  | jsignpdf --enable-stdin-passwords \
      -ksf keystore.p12 -ksp - -ka mykey \
      -ts https://tsa.example/ -tsu tsauser -tsp - \
      -d out/ input.pdf
```

Note the fixed order from §3.2: keystore password first, TSA password second, even though on the CLI they appear in the other order.

**Interactive:**

```sh
jsignpdf --enable-stdin-passwords -ksf keystore.p12 -ksp - -ka mykey input.pdf
# Enter password for --keystore-password: ****
```

## 8. Risk and rollback

- **Backwards compatibility risk:** none for users who do not set the new flag. Users who were previously passing `-` as a literal password and who also start using the new flag will need to quote or change their password; this is documented in the help text and the error message explicitly names the flag.
- **Rollback:** the feature is isolated to `StdinPasswordReader` + the parser refactor in `SignerOptionsFromCmdLine`. Reverting the PR fully removes it without data migrations.

## 10. Alternatives considered (not adopted)

- **Per-field source prefixes** (`-ksp env:KS_PWD`, `-ksp file:/run/secrets/ks`, `-ksp stdin`). More general and avoids the `-` collision, but introduces a new mini-DSL on every password option. Rejected for 3.0.0 in favor of the narrower, issue-#7-shaped change.
- **Environment variables** (`JSIGNPDF_KS_PASSWORD`, …). Clean for daemons but leaks through `/proc/<pid>/environ` to the same-uid peers and does not cover the interactive case. Can be added later as a complementary mechanism without conflicting with this design.
- **Single `--password-file`** with a properties-like format. Already approximated by `-lpf`; adding a second, passwords-only file felt redundant.
