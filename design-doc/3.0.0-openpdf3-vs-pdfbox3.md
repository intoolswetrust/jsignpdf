# OpenPDF 3 vs PDFBox 3 as the JSignPdf signing library

## Why this document exists

JSignPdf 2.x is built on **OpenPDF 1.3.30**, which is itself a fork of iText
2.1.7. The signing path (`SignerLogic.signFile()`,
`jsignpdf/src/main/java/net/sf/jsignpdf/SignerLogic.java`) is almost entirely
a direct call sequence against iText's classic signing API:
`PdfReader` → `PdfStamper.createSignature(...)` →
`PdfSignatureAppearance` → `PdfPKCS7` → `TSAClientBouncyCastle` /
`OcspClientBouncyCastle`.

PDFBox is only used for **preview rendering** (`preview/Pdf2Image.java`) and
as a **test validator** (`signing/validation/PdfSignatureValidator.java`).
It is not currently on the signing path.

The question this doc answers: if we ever want to leave the iText-2.1 lineage
— because OpenPDF 1.3.x is frozen, or because we want PAdES-LTV — what does
that migration look like? This is a research report, not a proposal to
migrate today.

## Candidates at a glance

| | **OpenPDF 3.x** | **PDFBox 3.x** |
|---|---|---|
| Current stable | 3.0.3 (Mar 2026) | 3.0.7 (Mar 2026) |
| Previous LTS-ish line | 2.2.5 (Mar 2026, Java 17), 1.3.43 (Mar 2024, Java 8, frozen) | 2.0.36 (Mar 2026, Java 8, still maintained) |
| Minimum Java (3.x) | **Java 21** | **Java 8** |
| Minimum Java (2.x) | **Java 17** (2.0.x) / **Java 21** (2.1.x+) | Java 8 |
| Lineage | Fork of iText 2.1.7 | Clean-sheet Apache project |
| License | LGPL 2.1 / MPL 2.0 | Apache 2.0 |
| 3.x package | `org.openpdf.*` (renamed from `com.lowagie.*`) | `org.apache.pdfbox.*` (unchanged) |
| Primary signing entry | `PdfStamper.createSignature(...)` + `PdfSignatureAppearance` (high-level) | `PDDocument.addSignature(PDSignature, SignatureInterface)` + you compose CMS (low-level) |
| BouncyCastle flavour | `bcprov-jdk18on` / `bcpkix-jdk18on` | `bcprov-jdk18on` / `bcpkix-jdk18on` (transitive, project-overridable) |

**Hard constraint — Java floor.** JSignPdf currently targets Java 11
(`maven.compiler.release=11`, enforcer rule `[11,12)` in the release
profile). Adopting OpenPDF 3.x forces Java 21; even OpenPDF 2.0.x forces
Java 17. OpenPDF 1.4.x (Java 11) exists but its last release was 1.4.2 in
April 2024 — effectively frozen. PDFBox 3.x keeps the Java 8 baseline,
so it stays comfortably under our Java 11 floor.

## Feature-by-feature comparison against JSignPdf 2.x

The rows below enumerate **every signing/encryption capability
`SignerLogic` exercises today**. "Direct" means the library ships a
high-level API that maps one-to-one to what JSignPdf does. "Manual" means
the capability is reachable but the caller has to assemble it (often with
BouncyCastle). "Missing" means the library doesn't offer it out of the
box.

| JSignPdf feature | Where in the current code | OpenPDF 3.x | PDFBox 3.x |
|---|---|---|---|
| Core CMS signing (`adbe.pkcs7.detached`) | `PdfPKCS7` at `SignerLogic.java:376–437` | **Direct** — identical API, `org.openpdf.text.pdf.PdfPKCS7` | **Manual** — implement `SignatureInterface.sign(InputStream)` and return a `CMSSignedData` byte[] assembled with BouncyCastle |
| Signature dictionary (`/Filter Adobe.PPKLite`) | `new PdfSignature(PdfName.ADOBE_PPKLITE, ...)` | **Direct** | **Direct** — `PDSignature.setFilter(FILTER_ADOBE_PPKLITE)` |
| Hash algorithms: SHA-1 / SHA-256 / SHA-384 / SHA-512 / RIPEMD-160 | `HashAlgorithm` enum + `PdfPKCS7(... hashAlgorithm, provider ...)` | **Direct** (all 5, provider-selectable) | **Manual** — digest is chosen inside `SignatureInterface.sign(...)`; BouncyCastle supports all 5 |
| Minimum-PDF-version bump for hash | `SignerLogic.java:187–206` (PDF 1.3 / 1.6 / 1.7 floor) | **Direct** — `PdfStamper.createSignature(reader, fout, tmpPdfVersion, ...)` (note: PDF-version parameter type changed from `char` to `String` in 3.0) | **Manual** — `PDDocument.setVersion(float)`; no hash/version matrix built in |
| Append mode / incremental save (append=true) | `PdfStamper.createSignature(..., append=true)` | **Direct** — same method signature | **Direct** — `PDDocument.saveIncremental(...)` / `saveIncrementalForExternalSigning(...)` |
| Remove existing signature fields (non-append) | `acroFields.removeField(sigName)` at `SignerLogic.java:212–218` | **Direct** — `AcroFields.removeField(...)` carried over | **Direct** — `PDAcroForm.getFields().remove(...)` (manual but straightforward) |
| Certification level / DocMDP (NOT_CERTIFIED, NO_CHANGES, FORM_FILLING, FORM_FILLING_AND_ANNOTATIONS) | `PdfSignatureAppearance.setCertificationLevel(int)` | **Direct** — same constants | **Direct** — `SignatureOptions.setPage(...)` + manual DocMDP entry; examples in `pdfbox/examples/signature/` cover all 4 levels |
| Signature metadata: reason / location / contact / date | `sap.setReason/setLocation/setContact`, `PdfDate` | **Direct** | **Direct** — `PDSignature.setReason/setLocation/setContactInfo/setSignDate` |
| Visible signature — rectangle + page | `sap.setVisibleSignature(rect, page, null)` | **Direct** | **Direct** — `PDVisibleSignDesigner` + `PDVisibleSigProperties` (constructors accept page, width/height, coordinates) |
| Visible signature — render mode (description-only / graphic+description / name+description) | `PdfSignatureAppearance.setRender(int)` via `RenderMode` | **Direct** — three modes preserved | **Missing as a mode flag** — PDFBox expects you to draw the appearance form XObject yourself (or via `PDVisibleSignDesigner`, which is image-oriented and has no "name-and-description" preset) |
| Visible signature — foreground/signature graphic | `sap.setSignatureGraphic(img)` | **Direct** | **Direct** — `PDVisibleSignDesigner.image(...)` |
| Visible signature — background image + scale | `sap.setImage(img)` + `setImageScale(float)` | **Direct** | **Manual** — has to be composed into the appearance stream |
| Visible signature — Layer 2 text with `${signer}/${certificate}/${timestamp}/${location}/${reason}/${contact}` placeholders | `sap.setLayer2Text(...)` with `StrSubstitutor` at `SignerLogic.java:307–326` | **Direct** — the layered appearance model (layers 0–4) is preserved; JSignPdf's placeholder substitution is its own code and is library-agnostic | **Missing** — PDFBox has no `layer2`/`layer4` concept; "Adobe Digital Signature Appearances" layers n0–n4 are not an API, they're a convention you'd have to implement. (Notable quote from PDFBox developers: "the n2 layer isn't supported in PDFBox; it can only work with image or invisible signatures".) |
| Visible signature — Layer 4 text | `sap.setLayer4Text(...)` | **Direct** | **Missing** (see above) |
| Visible signature — custom L2 font | `sap.setLayer2Font(new Font(baseFont, size))` + `FontUtils` | **Direct** | **Manual** — would be drawn via `PDPageContentStream` + `PDType0Font.load(...)` |
| Acrobat 6 layers toggle | `sap.setAcro6Layers(bool)` | **Direct** | **Not applicable** — PDFBox doesn't emit the iText-style multi-layer XObject regardless |
| PDF password encryption (user + owner + rights bitmap) at sign time | `stp.setEncryption(true, userPwd, ownerPwd, rights)` at `SignerLogic.java:229–238` | **Direct** — same `PdfStamper.setEncryption(...)` signature | **Direct** — `StandardProtectionPolicy` + `AccessPermission` + `PDDocument.protect(policy)` (note: **not** with incremental save — encryption has to be applied on a full save, which is at odds with detached signing. PDFBox 3 explicitly forbids using the input file as the output destination.) |
| Certificate-based encryption (recipient X.509, AES-128) | `stp.setEncryption(Certificate[], int[], ENCRYPTION_AES_128)` | **Direct** | **Direct** — `PublicKeyProtectionPolicy` + `PublicKeyRecipient.setX509(...)` + key length 128/256 |
| Permissions: print / degraded-print / copy / assembly / fill-in / screen-readers / modify-annotations / modify-contents | `BasicSignerOptions.getRightPrinting()/isRightCopy()/...` composed into bitmask | **Direct** — same bitmask constants on `PdfWriter` | **Direct** — equivalent setters on `AccessPermission` |
| Opening password-protected input | `new PdfReader(inFile, ownerPwd.getBytes())` with two-level fallback | **Direct** | **Direct** — `Loader.loadPDF(RandomAccessRead, String password)` |
| TSA (RFC 3161) timestamp embedding | `TSAClientBouncyCastle` at `SignerLogic.java:417–436` | **Direct** — `org.openpdf.text.pdf.TSAClientBouncyCastle` carried over; supports URL, basic-auth user/password, digest name, policy OID, proxy | **Partial** — `ExternalSigningSupport` + `saveIncrementalForExternalSigning`; the examples bundle a `ValidationTimeStamp` helper but there is no public `TSAClient` with digest/policy/basic-auth built in — JSignPdf would inherit ownership of the TSA HTTP client |
| OCSP response embedding in the PKCS#7 container | `OcspClientBouncyCastle(...)` → `PdfPKCS7.getEncodedPKCS7(..., ocsp)` at `SignerLogic.java:387–402,437` | **Direct** — `org.openpdf.text.pdf.OcspClientBouncyCastle` carried over (URL from cert AIA, proxy support) | **Partial** — PDFBox has OCSP validation helpers; embedding inside the signature CMS is still the caller's responsibility (BouncyCastle `OCSPResp` + CMS unsigned attributes) |
| CRL download + embedding | `CRLInfo` class, `PdfPKCS7(... crls, ...)` | **Direct** — `PdfPKCS7` accepts `CRL[]` at construction | **Manual** — build the CRL list yourself; embed via unsigned attributes or the DSS dictionary |
| External signing (CloudFoxy remote smartcard) | `PdfPKCS7.setExternalDigest(signature, null, "RSA")` | **Direct** — same API preserved | **Direct-ish** — PDFBox's `SignatureInterface`/`ExternalSigningSupport` is a better fit (was designed for this), but JSignPdf's existing CloudFoxy wiring would need to be rewritten to return a `CMSSignedData` rather than just signed bytes |
| Local PKCS#11 hardware token (`sun.security.pkcs11.SunPKCS11`) | `PKCS11Utils.getProviderNameForKeystoreType(...)` passed into `PdfPKCS7(...)` | **Direct** — provider string passed through iText signing | **Direct** — `KeyStore.getInstance("PKCS11", provider)` → `PrivateKey` handed to `SignatureInterface.sign(...)`; fully documented pattern |
| PAdES (`ETSI.CAdES.detached` subfilter) | Not currently implemented (open feature request #46) | **Missing** — OpenPDF issue #86 "PAdES signatures support" is closed **wontfix**; PdfPKCS7 only emits `adbe.pkcs7.*` subfilters | **Direct** — PDFBox produces all three CAdES-based subfilters, and the `SwisscomTrustServices/pdfbox-ais-client` and `esig/dss` projects both use PDFBox for PAdES |
| PAdES B-T / B-LT / B-LTA — DSS dictionary, VRI, document timestamp | Not implemented (JSignPdf #46) | **Missing** — no DSS / VRI support in OpenPDF | **Direct** — `PDDocument.addSignature(...)` with a document-timestamp `PDSignature` and the DSS entries under `/Catalog/DSS`; example code exists upstream |
| Proxy support (TSA / OCSP / CRL) | `options.createProxy()` → `tsc.setProxy(proxy)` etc. | **Direct** — iText clients take `java.net.Proxy` | **Missing at API level** — the TSA/OCSP code you'd write would need to honour the proxy yourself |

### Summary of the table

- **OpenPDF 3.x is effectively API-compatible with the current JSignPdf
  code.** The 3.0.0 release is mostly a package rename (`com.lowagie` →
  `org.openpdf`) plus a `char → String` flip on PDF-version methods. Every
  signing-related class JSignPdf uses today (`PdfPKCS7`,
  `PdfSignatureAppearance`, `PdfStamper`, `AcroFields`, `PdfReader`,
  `PdfSignature`, `TSAClientBouncyCastle`, `OcspClientBouncyCastle`,
  `PdfName.ADOBE_PPKLITE`) is carried forward. The migration is
  mechanical.
- **PDFBox 3.x covers all the _cryptographic_ primitives but asks the
  caller to assemble them.** The level-of-abstraction gap is concentrated
  in the visible-signature appearance and the TSA/OCSP/CRL embedding
  helpers — exactly the parts of iText 2.1 that made it popular for PDF
  signing fifteen years ago. We would move from ~30 lines of
  `PdfSignatureAppearance.setLayer2Text(...)` to ~300 lines of our own
  layered-XObject composer.
- **PDFBox 3 uniquely unlocks PAdES-LTV.** If JSignPdf issue #46 (B-LTA
  support) ever becomes a requirement, OpenPDF cannot get us there —
  upstream is explicit about this. PDFBox is the only realistic path.

## Non-feature dimensions

| Dimension | OpenPDF 3.x | PDFBox 3.x |
|---|---|---|
| Upstream velocity | Small core team; ~87 releases overall, cadence of ~1/month; responsive on GitHub | Apache project; corporate contributors; monthly point releases on 3.x, parallel maintenance on 2.x |
| Signing hardening history | Recent CVEs/Issues around signature wrapping (issue #487); iText-2.1 codebase, age shows | Mature signing-specific code path; extensive test suite in `pdfbox-examples/signature` |
| Licence fit for JSignPdf (MPL 2.0 + LGPL 2.1 dual) | **Matching** — OpenPDF is LGPL 2.1/MPL 2.0 | **Permissive (Apache 2.0)** — compatible, but changes the overall licence story of the signing core |
| Artifact size (fat-jar) | Small core (~2 MB) | Larger (pdfbox + fontbox + pdfbox-io ≈ 8–10 MB) — but we already bundle PDFBox for the preview and validator path |
| Documentation for signing | Sparse. JavaDoc on the old iText names; community answers are mostly iText/OpenPDF crossovers. | Extensive. Upstream ships `examples/signature/CreateSignature*.java`, `CreateVisibleSignature*.java`, PAdES-LTV examples in third-party projects |
| Dependency on BouncyCastle | Optional but effectively required for signing; project already ships BC 1.84 | Required; 3.x tracks BC 1.75+ transitively (project can override) |

## Trade-off summary

**Option A — OpenPDF 3.x (direct port of today's code).**
- Cheap to do *only if* we accept the Java 21 floor. That is the gating
  concern: JSignPdf today supports Java 11 and we bundle JREs with
  jpackage. Jumping to Java 21 is a packaging/support decision, not a
  library decision.
- If the Java-21 jump is acceptable, the code migration is a find/replace
  of imports plus `char` → `String` in a handful of call sites, plus a
  few OpenPDF 2.x API stabilisations to absorb (e.g. any deprecations we
  inherited from iText 2.1).
- Gets us **no new features**. In particular, no PAdES, no LTV.
- Leaves us on the ageing iText-2.1 signing pipeline for another decade.

**Option A′ — Stay on OpenPDF 1.3.x.**
- Status quo. Branch is frozen (last release Mar 2024). Security fixes,
  if any are ever needed, would require us to fork.
- Zero work, growing tech-debt curve.

**Option B — PDFBox 3.x (rewrite `SignerLogic` on top of PDFBox
signing).**
- Keeps Java 11 floor. Already bundled (preview + test validator), so
  no new top-level dependency — though the shaded-jar composition changes
  because PDFBox moves onto the runtime signing path.
- Substantial engineering work: visible-signature appearance composer,
  a TSA client that honours our current feature set (policy OID,
  basic-auth, proxy), an OCSP embedder, and a migration of
  `BasicSignerOptions`-driven placeholders onto a custom appearance
  builder. Estimate: a 2–4 week focused effort with careful test-corpus
  coverage, not a weekend refactor.
- Unlocks PAdES B / B-T / B-LT / B-LTA in a way OpenPDF can't.
- Aligns JSignPdf with the codebase the EU DSS project and the Swisscom
  AIS client have already battle-tested for signing.

**Option C — Hybrid (compose both).**
- Keep OpenPDF for the current iText-style path; add a new PDFBox-backed
  signing strategy behind a feature flag (e.g. `signingEngine=pdfbox` or
  driven by `padesLevel=B-LTA`). User-visible options from
  `BasicSignerOptions` that don't translate (layered L2/L4 text, Acro6
  layers) are fenced off when the PDFBox engine is selected.
- Highest-complexity option but leaves existing users untouched and lets
  us ship PAdES as an opt-in. Bundled-jar size grows, but since we
  already ship PDFBox, the delta is modest.

## Recommendation (for future decision-making, not for now)

Don't migrate today. The current OpenPDF 1.3.30 path works, covers every
feature JSignPdf advertises, and is understood.

When a trigger appears — specifically **PAdES-LTV demand** (#46) or a
**signature-wrapping CVE in OpenPDF** — the right path is **Option C**:
introduce a parallel PDFBox-backed signing strategy rather than doing a
full port. That choice:

- doesn't force a Java version bump (unlike OpenPDF 2.x/3.x),
- doesn't destabilise signature-appearance output for existing users
  (layer-based L2/L4 text continues to work exactly as before),
- buys us PAdES without us taking on the maintenance burden of a custom
  PAdES-on-OpenPDF patch set,
- uses a library we already ship.

Parking this report here so the analysis is ready when that trigger
arrives.

## What this document is not

- Not a migration plan. A real migration plan would enumerate the
  `SignerLogic` call sites, the `SignPdfForm` UI fields that map to
  visible-signature options, the CLI flag compatibility matrix, and the
  test-corpus delta. That belongs in a separate doc if/when we decide
  to proceed.
- Not a security review of either library. See the upstream CVE history
  and any downstream advisories before committing.
- Not an evaluation of non-signing features (HTML→PDF, text extraction,
  form filling). JSignPdf only signs.
