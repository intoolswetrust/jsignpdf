# Pluggable signing engines

## Why this document exists

Today the entire signing path is hard-coded to OpenPDF 1.3 (`SignerLogic.signFile()`,
`jsignpdf/src/main/java/net/sf/jsignpdf/SignerLogic.java`). That path uses
`PdfStamper.createSignature(...)` → `PdfSignatureAppearance` → `PdfPKCS7` and a
chain of iText-2.1 derived helpers. PDFBox is on the classpath but only for
preview rendering and as a test validator — it is not on the signing path.

`design-doc/3.0.0-openpdf3-vs-pdfbox3.md` already analysed the trade-offs of
swapping the signing library. Its recommendation was *not* a wholesale port,
but a **hybrid model**: keep OpenPDF for what it does well, add additional
backends behind a flag for what it doesn't (notably PAdES-LTV via PDFBox or
the EU DSS project).

This document is the design for that hybrid model. It introduces a
**signing engine abstraction**, makes engines pluggable via `ServiceLoader`,
and lays out a two-phase rollout:

- **Phase 1** — introduce the abstraction, no behaviour change. Today's
  OpenPDF code moves into a dedicated `jsignpdf-engine-openpdf` module
  behind the new API. Default engine is `openpdf`. Existing config
  files keep working unchanged.
- **Phase 2** — add `jsignpdf-engine-pdfbox` and `jsignpdf-engine-dss`
  modules (each in its own PR / design doc). The engine API gets
  iterated in light of the two new implementations.

## Goals and non-goals

**Goals.**

- Make the signing backend selectable at runtime, per-invocation.
- Expose engine capabilities through a typed API so the CLI can fail
  fast and the FX UI can disable controls the active engine doesn't
  support.
- Allow third parties to drop a jar into the `lib/` directory of an
  installed JSignPdf and have a new engine appear in `--list-engines`
  / the FX dropdown automatically (Maven-level pluggability).
- Keep phase 1 a pure refactor: identical signature output, identical
  behaviour, no user-visible regressions for existing flows.

**Non-goals.**

- A plugin browser / "install engine X from the internet" UI.
  Pluggability is at the file-system / Maven level only.
- Per-capability i18n strings. The FX UI uses one shared "not
  supported by the selected engine" tooltip. Per-capability tooltips
  can be added later without API changes.
- Engine-aware `--help` (filtered or annotated). `--help` keeps
  showing every option; the validator surfaces engine mismatches at
  sign time.
- Reworking `BasicSignerOptions` into a richer model. It stays a
  property bag; new fields are added only when a shipping engine
  genuinely needs them.
- Designing the cross-engine external-signing API now (CloudFoxy).
  Phase 1 keeps CloudFoxy inside the OpenPDF engine; the abstraction
  is revisited in phase 2 with PDFBox and DSS in front of us.

## Phase plan at a glance

| | **Phase 1 (this doc)** | **Phase 2 (own design docs)** |
|---|---|---|
| Modules added | `engine-api`, `engine-openpdf` | `engine-pdfbox`, `engine-dss` |
| Default engine | `openpdf` | `openpdf` (unchanged) |
| Capabilities exposed | Whatever OpenPDF already does | New `PADES_BASELINE_*`, `DSS_DICTIONARY`, `SUBFILTER_ETSI_CADES_DETACHED` |
| `BasicSignerOptions` changes | New `engine` field only | DSS adds `padesLevel` (and similar) |
| CLI changes | `-eng / --engine`, `-le / --list-engines` | None expected |
| FX UI changes | Engine dropdown in toolbar; capability-driven disable | None structural; new options bind to existing pattern |
| External signing (CloudFoxy) | Stays inside OpenPDF engine | Engine API gets revisited; cross-engine support designed |
| Distribution | One bundle, includes OpenPDF engine | One bundle, includes all three engines |
| User-visible regressions | None expected | None expected |

## Module layout

The repo currently has a flat top-level: `jsignpdf/`, `installcert/`,
`distribution/`. We add an `engines/` aggregator under the root pom:

```
jsignpdf-root/
├── jsignpdf/                       # main app (signing dispatcher + GUI + CLI)
├── engines/
│   ├── pom.xml                     # aggregator; pulls in api + each engine
│   ├── api/                        # artifactId: jsignpdf-engine-api
│   ├── openpdf/                    # artifactId: jsignpdf-engine-openpdf  (phase 1)
│   ├── pdfbox/                     # artifactId: jsignpdf-engine-pdfbox   (phase 2)
│   └── dss/                        # artifactId: jsignpdf-engine-dss      (phase 2)
├── installcert/
└── distribution/
```

ArtifactIds: `jsignpdf-engine-api`, `jsignpdf-engine-openpdf`,
`jsignpdf-engine-pdfbox`, `jsignpdf-engine-dss`. GroupId remains
`com.github.kwart.jsign`. Folder names are short (`api`, `openpdf`, …)
since the `engines/` parent already disambiguates them; Maven artifactIds
stay fully qualified for clarity in dependency declarations.

The `engines/pom.xml` aggregator centralises shared plugin / property
config for all engines (e.g. shaded jar conventions, surefire setup) so
each engine module's pom stays small.

## Engine API

The new `jsignpdf-engine-api` module is a thin SPI. It depends only on
`bcprov-jdk18on` / `bcpkix-jdk18on` (for shared crypto types) and
whatever shared JSignPdf internals the engines genuinely need
(`BasicSignerOptions`, `HashAlgorithm`, etc.). It does **not** depend on
OpenPDF, PDFBox, or DSS.

> Because `BasicSignerOptions` lives in the main `jsignpdf` module
> today, phase 1 also moves it (plus its associated value types under
> `net.sf.jsignpdf.types`, the listener interfaces, and a few shared
> utilities like `SSLInitializer`) either into the API module or into a
> small `engine-api`-adjacent core jar. Concretely: phase 1 introduces a
> new `net.sf.jsignpdf.engine` package in the API module and either
> (a) moves `BasicSignerOptions` and the value types there, or
> (b) leaves them in the main module and the API module declares a
> compile-time dependency on the main module. Option (a) is cleaner;
> option (b) is the smaller diff. Picked at implementation time —
> not load-bearing for any decision in this doc.

Core interface:

```java
package net.sf.jsignpdf.engine;

public interface SigningEngine {

    /** Stable, lower-case identifier used in config files and CLI args. */
    String id();

    /** Human-readable name shown in --list-engines and the FX dropdown. */
    String displayName();

    /**
     * Static capability set for this engine. Returned set is immutable
     * and identical across all calls / instances of the same engine.
     */
    Set<Capability> capabilities();

    /**
     * Signs the file described by options. Implementation owns its
     * own logging via the same SLF4J/JUL bridge the rest of JSignPdf
     * uses. Returns true on success, false on a recoverable error
     * (already-logged); throws for unrecoverable problems.
     */
    boolean sign(BasicSignerOptions options, EngineConfig engineConfig);
}
```

`EngineConfig` is a thin read-only view of `advanced.properties`, scoped
to the active engine's `engine.<id>.*` namespace:

```java
public interface EngineConfig {
    String getString(String key);                   // null if absent
    String getString(String key, String fallback);
    boolean getBoolean(String key, boolean fallback);
    int getInt(String key, int fallback);
}
```

Engines read their own knobs through `EngineConfig` (e.g. DSS will read
`getString("padesLevel", "B")` which under the hood resolves
`engine.dss.padesLevel`). The main module passes the right namespaced
view in.

The `EngineRegistry` (in the main `jsignpdf` module, **not** in the API)
is the lookup point:

```java
public final class EngineRegistry {
    public static EngineRegistry getInstance();
    public Optional<SigningEngine> findById(String id);
    public List<SigningEngine> listAll();      // sorted: default first, then by id
    public SigningEngine getDefault();         // resolves advanced.properties → "openpdf"
}
```

Internally the registry uses
`ServiceLoader.load(SigningEngine.class)`. Engines on the classpath
register themselves via `META-INF/services/net.sf.jsignpdf.engine.SigningEngine`.
The registry caches the result for the JVM lifetime. Duplicate ids are
detected at registry construction time and logged as a warning; the
first-loaded entry wins (deterministic by classpath order).

### Discovery and shading

The default distribution is a shaded fat-jar built by
`maven-shade-plugin`. To preserve `META-INF/services` entries on shade
we add the standard transformer to the shade configuration:

```xml
<transformer implementation="org.apache.maven.plugins.shade.resource.ServicesResourceTransformer"/>
```

This is a known one-line fix. Without it, all engine SPI files would
collide on `META-INF/services/net.sf.jsignpdf.engine.SigningEngine` and
only one engine would survive shading. Tested by an integration test
that asserts `--list-engines` lists every engine bundled in the
distribution.

The same transformer behaviour is needed for the jpackage-installer
build path; verified during phase 1 implementation.

## Capability model

A single fine-grained `Capability` enum, declared in `engine-api`. Each
engine returns a static `EnumSet<Capability>` from `capabilities()`.
Absent = unsupported.

```java
public enum Capability {

    // signature container / format
    SUBFILTER_ADBE_PKCS7_DETACHED,
    SUBFILTER_ETSI_CADES_DETACHED,             // PAdES baseline subfilter
    PADES_BASELINE_B,
    PADES_BASELINE_T,
    PADES_BASELINE_LT,
    PADES_BASELINE_LTA,

    // hashes
    HASH_SHA1, HASH_SHA256, HASH_SHA384, HASH_SHA512, HASH_RIPEMD160,

    // document-level
    APPEND_MODE,
    CERTIFICATION_LEVEL,                       // DocMDP: all four levels
    ENCRYPTION_PASSWORD,
    ENCRYPTION_CERTIFICATE,
    PERMISSIONS_BITMASK,                       // print / copy / assembly / ...

    // visible signature
    VISIBLE_SIGNATURE,
    VISIBLE_LAYER2_TEXT,                       // %signer / %timestamp / ...
    VISIBLE_LAYER4_TEXT,
    VISIBLE_RENDER_MODE_DESCRIPTION_ONLY,
    VISIBLE_RENDER_MODE_GRAPHIC_AND_DESCRIPTION,
    VISIBLE_RENDER_MODE_NAME_AND_DESCRIPTION,
    VISIBLE_BACKGROUND_IMAGE,
    VISIBLE_BACKGROUND_IMAGE_SCALE,
    VISIBLE_SIGNATURE_GRAPHIC,
    VISIBLE_CUSTOM_FONT,
    ACRO6_LAYERS,

    // revocation / timestamp
    TSA,
    TSA_POLICY_OID,
    TSA_BASIC_AUTH,
    OCSP_EMBED,
    CRL_EMBED,
    DSS_DICTIONARY,                            // PAdES-LT/LTA prerequisite

    // transport
    PROXY_SUPPORT,

    // keystores / external signing
    EXTERNAL_DIGEST,                           // CloudFoxy-style externally-produced signature
    PKCS11_PROVIDER
}
```

The OpenPDF engine in phase 1 returns a set that exactly matches
JSignPdf 2.x's current capabilities: every flag above except the
`PADES_BASELINE_*` family, `SUBFILTER_ETSI_CADES_DETACHED`, and
`DSS_DICTIONARY`.

The set is immutable per engine instance, identical across all
invocations of the same engine. (Dynamic capabilities — e.g. DSS that
varies its set based on the chosen PAdES baseline — are out of scope;
revisit if a real use case appears.)

No i18n is wired up for capability names in phase 1. The shared "not
supported" tooltip text is the only user-facing string.

## Configuration

Engine selection lives in `advanced.properties` (per
`design-doc/3.0.0-config-expose.md` and the existing `AppConfig`
two-layer overlay).

New keys in `advanced.default.properties` (the bundled fallback):

```
# Identifier of the signing engine used by default. Use --list-engines
# to see available identifiers. The user-config file
# <cfg>/advanced.properties may override this.
engine=openpdf

# Engine-specific configuration. Each engine module reads its own keys
# under the engine.<id>.* prefix. No keys are defined for the OpenPDF
# engine; future engines (PDFBox, DSS) will document their keys in
# their own design docs.
```

Resolution precedence (highest → lowest):

1. CLI flag `--engine <id>` (CLI-only flow).
2. `advanced.properties` `engine=<id>` (user override).
3. `advanced.default.properties` (bundled `engine=openpdf`).

The FX UI does **not** use the CLI flag (it's a CLI-only override). The
FX engine dropdown reads from and writes to the user's
`advanced.properties` `engine=` key directly.

Engine-specific app-global config (`engine.<id>.*`) is read by the
engine itself through `EngineConfig`. It is not surfaced in the
Preferences dialog in phase 1 (no engine has any such keys yet);
phase 2 (DSS in particular) will extend the Preferences UI as needed.

Per-document engine-specific fields (e.g. PAdES level) — when added in
phase 2 — go on `BasicSignerOptions` and are persisted in `config.properties`
and presets the same way every other field is. Engines that don't
understand them ignore them silently; the validator only complains if
the user *explicitly* set such a field while running an engine that
doesn't support it.

## CLI changes

Two new options, registered in the same place as the existing
`--list-keystore-types` / `--keystore-type` pair (`Constants.java`,
`SignerOptionsFromCmdLine.java`).

| Short | Long | Argument | Behaviour |
|---|---|---|---|
| `-eng` | `--engine` | engine id | Overrides `advanced.properties` for this invocation. |
| `-le` | `--list-engines` | none | Prints engines, one per line, in the form `<id> - <displayName>`. The default engine is annotated `(default)` at the end of its line. Exits with code 0. |

`--help` keeps showing every option; engine annotations are deliberately
not added to keep the help text stable and the implementation simple.

Engine-mismatch validation runs **before** any I/O at sign time: after
`SignerOptionsFromCmdLine` has populated `BasicSignerOptions` and the
engine has been resolved, the dispatcher walks the populated fields and
checks each non-default value against the active engine's capability
set. Any mismatch produces a single error message listing every
offending option and the active engine, and exits non-zero
(`Constants.EXIT_CODE_PARSE_ERROR` or a new `EXIT_CODE_ENGINE_MISMATCH`
— picked at implementation time).

Example:

```
$ jsignpdf --engine pdfbox -L "%signer\n%timestamp" foo.pdf
ERROR: The selected engine 'pdfbox' does not support the following options:
  --l2-text         (capability VISIBLE_LAYER2_TEXT)
Use --list-engines to see available engines.
```

(Phase 1 has only the OpenPDF engine, so this error path is exercisable
end-to-end only via tests that spin up a fake stub engine. It is
nevertheless implemented and covered by tests in phase 1 so phase 2
inherits a working validator.)

## FX UI changes

### Engine selector

A new `ChoiceBox<SigningEngine>` is added to `MainWindowController`'s
toolbar (top of the main window). Items come from
`EngineRegistry.listAll()`; selection is bound bidirectionally to the
`engine=` key in `advanced.properties`. Cell renderer shows
`displayName()`.

Switching the engine is immediate — no restart, no save dialog. The
FX layer reacts by re-evaluating capability bindings on every panel.

### Capability-driven enable/disable

Each FX control that maps to an engine capability binds its
`disableProperty` to a `BooleanBinding` derived from the active engine:

```java
BooleanBinding unsupported = Bindings.createBooleanBinding(
    () -> !engineProperty.get().capabilities().contains(Capability.VISIBLE_LAYER2_TEXT),
    engineProperty);
l2TextArea.disableProperty().bind(unsupported);
l2TextArea.tooltipProperty().bind(
    Bindings.when(unsupported).then(unsupportedTooltip).otherwise((Tooltip) null));
```

Two granularity rules:

- **Section-level when an umbrella capability is missing.** If the
  active engine doesn't declare `VISIBLE_SIGNATURE`, the entire
  "Visible signature" tab/section is disabled and its sub-controls are
  not individually evaluated. Same pattern for any future umbrella
  capability.
- **Field-level inside a supported section.** If `VISIBLE_SIGNATURE`
  is supported but `VISIBLE_LAYER2_TEXT` isn't, the section stays
  enabled and only the L2 text area is greyed out. PDFBox in phase 2
  is the first concrete case of this — it supports visible signatures
  but not the layered-text appearance model.

A single shared i18n key drives the tooltip text:

```
jfx.gui.engine.unsupported = This option is not supported by the selected engine.
```

(Per-capability tooltips can be added later by introducing
per-capability keys; the binding code stays the same.)

## Code migration plan (phase 1)

The current `SignerLogic.signFile()` in
`jsignpdf/src/main/java/net/sf/jsignpdf/SignerLogic.java` is ~380
lines, almost all OpenPDF-specific. The migration is mechanical:

**Stays in the `jsignpdf` main module:**
- The `signFile()` *dispatcher* — renamed/refactored to a thin shell
  that resolves the engine, builds the `EngineConfig` view,
  validates `BasicSignerOptions` against the engine's capabilities,
  and calls `engine.sign(options, engineConfig)`.
- File-existence validation (`validateInOutFiles`).
- Engine-agnostic helpers (e.g. proxy construction from `Options`,
  if any of those are not engine-specific).
- All FX / Swing UI code (no signing-library imports here today).
- `KeyStoreUtils`, `PKCS11Utils`, `PropertyProvider`, `AppConfig`,
  `ConfigLocationResolver` (all engine-agnostic).

**Moves to `jsignpdf-engine-openpdf`:**
- The body of today's `SignerLogic.signFile()`, lifted into an
  `OpenPdfSigningEngine implements SigningEngine`.
- All `org.openpdf.text.pdf.*` import sites — `PdfStamper`,
  `PdfReader`, `PdfPKCS7`, `PdfSignatureAppearance`, `AcroFields`,
  `TSAClientBouncyCastle`, `OcspClientBouncyCastle`, …
- `net.sf.jsignpdf.crl.CRLInfo` — only used from the OpenPDF signing
  path today.
- `net.sf.jsignpdf.extcsp.CloudFoxy` (see "CloudFoxy" section below).
- The relevant pieces of `FontUtils` that touch
  `org.openpdf.text.pdf.BaseFont`. (FontUtils may end up split: the
  config-resolution side stays in main; the OpenPDF-specific
  `getL2BaseFont()` moves.)
- The `META-INF/services/net.sf.jsignpdf.engine.SigningEngine`
  registration file.

**Moves to `jsignpdf-engine-api`:**
- New: `SigningEngine`, `Capability`, `EngineConfig` interfaces.
- `BasicSignerOptions` and the `net.sf.jsignpdf.types.*` value objects
  (`HashAlgorithm`, `RenderMode`, `PdfVersion`, `CertificationLevel`,
  `PDFEncryption`, `PrintRight`, `ServerAuthentication`) — they're
  shared vocabulary between engines.
- Listener interfaces (`SignResultListener`).

The dispatcher's call site changes from:

```java
new SignerLogic(options).signFile();
```

to (sketch):

```java
SigningEngine engine = EngineRegistry.getInstance()
        .findById(resolveEngineId(options))
        .orElseThrow(() -> new RuntimeException("Unknown engine"));

EngineMismatchValidator.validate(options, engine);     // throws on mismatch

EngineConfig engineConfig = AppConfig.engineConfigFor(engine.id());
engine.sign(options, engineConfig);
```

Output guarantee: the OpenPDF engine in phase 1 produces signatures
**byte-identical** to today's `SignerLogic` output (modulo timestamps
and the PRNG-driven nonce in CMS, which were already non-deterministic
before). Tested by replaying the existing `signing/` test corpus.

## CloudFoxy

CloudFoxy support is currently special-cased in `SignerLogic.java` at
two points: the keystore-source branch (line 118) and the
external-signature branch (line 377). It is modelled today as a
"keystore type" rather than an engine-level concern.

For phase 1:

- `net.sf.jsignpdf.extcsp.CloudFoxy` and its supporting classes move
  into `jsignpdf-engine-openpdf`. The OpenPDF engine internally
  retains the special-case branch as it exists today.
- The OpenPDF engine declares `EXTERNAL_DIGEST` in its capability set.
- PDFBox and DSS engines (phase 2) initially do **not** declare
  `EXTERNAL_DIGEST` and therefore reject `KEYSTORE_TYPE_CLOUDFOXY` via
  the engine-mismatch validator. Switching the engine to OpenPDF makes
  the option valid again.
- The cross-engine external-signing API is **deliberately not designed
  in phase 1**. PDFBox's `ExternalSigningSupport` and DSS's
  `SignatureToken` model are different enough from a `byte[]→byte[]`
  function that designing the abstraction in a vacuum risks a
  premature shape. Phase 2 designs it with three real implementations
  in front of it; the resulting API change is acceptable since the
  engine API surface is still young and only in-tree engines exist.

## Distribution and build

The current distribution is a shaded fat-jar plus a `lib/` directory
with non-shaded extras (per `distribution/`). With pluggable engines:

- The shaded jar contains `engine-api`, `engine-openpdf` (phase 1)
  and additionally `engine-pdfbox`, `engine-dss` (phase 2). Shading
  preserves all `META-INF/services` entries via
  `ServicesResourceTransformer`.
- All three engines are bundled in the default distribution from
  phase 2 onward. The bundle-size impact (PDFBox ~8–10 MB already
  present for preview; DSS ~30+ MB new) is acknowledged in the phase-2
  release notes. Trade-off accepted in favour of a single,
  one-download experience.
- LGPL implication: DSS is LGPL 2.1. JSignPdf is dual MPL 2.0 / LGPL
  2.1, so license-compatible. Bundling DSS does, however, put an LGPL
  dependency on the active signing path that wasn't there before;
  flagged in the phase-2 design doc and release notes.
- Pluggability path for advanced users: a third-party `SigningEngine`
  jar (with its dependencies and a `META-INF/services` registration)
  dropped into the installed `lib/` directory will be picked up by the
  `ServiceLoader` at startup, surfaced in `--list-engines` and the FX
  dropdown, and usable. This is implicitly tested by the in-tree
  engines themselves (each is "just another jar").

The jpackage installer build (`design-doc/3.0.0-jpackage-installer.md`)
needs to be re-checked: jpackage rolls the `lib/` directory into the
app image. A short check during phase 1 implementation confirms
service loading works inside the jpackaged image.

## Testing

**Phase 1 — additions:**

- `EngineRegistryTest`: verifies `ServiceLoader` picks up the OpenPDF
  engine; default resolution returns `openpdf`; duplicate-id detection
  logs and keeps the first.
- `EngineMismatchValidatorTest`: exercised with a fake stub
  `SigningEngine` that declares a deliberately small capability set;
  asserts every gateable field raises an error when set; asserts no
  errors when fields are at their default / unset values.
- `OpenPdfSigningEngineTest`: a thin wrapper that re-runs the existing
  `signing/` corpus through the engine API instead of `SignerLogic`
  directly; output bytes (sans nonces / dates) must match.
- `SignerOptionsFromCmdLineTest`: new cases for `--engine` and
  `--list-engines`.
- `FxTranslationsTest`: new keys covered by the existing translation
  presence assertions.
- Smoke test on the shaded distribution jar that
  `java -jar jsignpdf.jar --list-engines` lists `openpdf`.

**Phase 1 — unchanged:**

- All existing `signing/` tests run unchanged. Default engine is
  `openpdf`; signature output is byte-identical (modulo nonces /
  timestamps, as before).

**Phase 2 — to be detailed in each engine's own design doc.** Each
engine ships its own corpus; cross-engine determinism is not a goal.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| OpenPDF engine output drifts from current `SignerLogic` output during the move | Byte-equality tests against the existing corpus, run as part of CI before the phase-1 PR merges. |
| `ServicesResourceTransformer` not configured / silently dropped | Unit-test on the shaded jar that asserts engine count > 0. |
| jpackage installer breaks `ServiceLoader` discovery | Smoke-test the jpackage image (CI workflow already exists per `do-release.yml`). |
| Engine mismatch validator misses a field | Validator is table-driven: same lookup table used by the FX UI and the CLI. Adding a new option means adding a row in one place. |
| `BasicSignerOptions` move into `engine-api` ripples through every consumer | The move is optional (option (b) in the "Engine API" section keeps it where it is). Pick (a) only if the diff is manageable. |
| Phase 2 engine-API change for external signing breaks third-party engines | Acceptable — the API is explicitly young in phase 1; no third-party engines are expected before phase 2 lands. |

## Open items (not blocking phase 1)

- **`BasicSignerOptions` location.** Phase 1 implementation picks
  between (a) move to `engine-api` and (b) leave in main and have
  `engine-api` depend on main. Decided at implementation time based on
  diff size; not load-bearing for any decision in this doc.
- **Engine-specific config exposure in the Preferences dialog.** No
  engine has any `engine.<id>.*` keys in phase 1, so nothing is shown.
  Phase 2 (DSS) needs to extend `PreferencesController` for PAdES
  level; designed in its own doc.
- **Per-capability tooltip i18n.** Out of scope; can be added later
  without API changes by introducing per-capability keys in
  `messages.properties` and a small lookup in the binding code.
- **Cross-engine external-signing API.** Out of scope until phase 2,
  by explicit decision (see CloudFoxy section).

## Out of scope (explicit non-goals)

- Filtering / annotating `--help` output by the active engine.
- A user-facing "install engine X" UI.
- Reworking `BasicSignerOptions` into a richer / engine-aware model.
- Dynamic capability sets (capabilities that vary across instances of
  the same engine).
- Designing the cross-engine external-signing API in phase 1.
