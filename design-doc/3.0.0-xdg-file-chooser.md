# Native File Chooser on Linux Sandboxes (XDG Desktop Portal)

**Status:** design
**Scope:** JSignPdf 3.0.0 JavaFX UI, Linux only, activated inside Flatpak / Snap sandboxes.
**Authors:** design produced by a three-role team ‚Äî Linux/XDG expert ‚Üí JavaFX expert ‚Üí UX reviewer.

---

## Problem

The JavaFX UI uses `javafx.stage.FileChooser`, which on Linux calls GTK's dialog APIs directly. In Flatpak/Snap sandboxes this bypasses the XDG Desktop Portal and therefore the sandbox's host-file access controls. The Flathub submission [flathub/flathub#8338](https://github.com/flathub/flathub/pull/8338) currently only passes review with `--filesystem=home`, an explicit exception that reviewers (@CodedOre) asked us to replace with portal support. The umbrella issue is [intoolswetrust/jsignpdf#307](https://github.com/intoolswetrust/jsignpdf/issues/307).

## Goal

Make JSignPdf's file choosers use `org.freedesktop.portal.FileChooser` **when, and only when, we are running inside a Linux sandbox**. On Windows, macOS, and non-sandboxed Linux the JavaFX chooser stays. The change should be mechanically small at each call site and revertible in one commit.

## Non-goals

- No portal use outside sandboxes. Bare-metal Wayland is out of scope.
- No change to the Swing UI (still reachable via `-Djsignpdf.swing=true`).
- No new file types, no multi-select that didn't exist before.
- No xdg-foreign parent-window handle plumbing in v1 (documented as v2).
- No directory chooser (`showDirectoryDialog`) ‚Äî no current caller.
- No replacement of JavaFX's `FileChooser` on non-Linux OSes.

## Options considered and rejected

| Option | Why rejected |
|---|---|
| Stargate | Requires Java 25; JSignPdf just moved to 21. |
| FlatLaf system file chooser | Swing-only; JavaFX has no LAF seam. Per ChatGPT consultation: mixing Swing into JavaFX for one dialog is worse than the problem. |
| `gdbus` subprocess | Works but is fragile (scraping output, race on signal subscription, no `Request.Close`). ChatGPT's "clean DBus-native" advice prefers a real binding. |
| Async `CompletableFuture<File>` API at call sites | Would ripple into all 7 controllers. Blocking matches JavaFX's own `FileChooser` contract. |

## Linux / XDG Portal Background

The XDG Desktop Portal is a D-Bus service (`xdg-desktop-portal`) that mediates host-resource access for sandboxed apps. Each desktop environment ships a backend (`xdg-desktop-portal-gtk`, `-kde`, `-gnome`, `-hyprland`, etc.) that renders the actual file dialog in the host's toolkit. From the app's perspective it's a single D-Bus API regardless of backend.

### 1. Sandbox detection

| Signal | Scope | Notes |
|---|---|---|
| `/.flatpak-info` exists | Flatpak | **Authoritative** for Flatpak. Present inside the sandbox, absent outside. |
| `FLATPAK_ID` env var | Flatpak | Set by `flatpak run`, but can be inherited by subprocesses that escaped via `flatpak-spawn --host`. Prefer the file check. |
| `SNAP` env var | Snap | Points at the snap's mount root (e.g. `/snap/jsignpdf/current`). Authoritative for Snap. |
| `SNAP_NAME` | Snap | Secondary confirmation. |
| `container` env var | Generic | Set to `flatpak`, `snap`, `podman`, `oci`, etc. Indicates *any* container, not necessarily a portal-speaking sandbox. Use only as a tiebreaker. |

Recommended single check:

```java
static boolean isSandboxed() {
    return Files.exists(Path.of("/.flatpak-info"))
        || System.getenv("FLATPAK_ID") != null
        || System.getenv("SNAP") != null;
}
```

Everything else in this document is gated on this returning `true` **and** `os.name` being Linux.

### 2. Portal API surface

| | Value |
|---|---|
| Bus name | `org.freedesktop.portal.Desktop` |
| Object path | `/org/freedesktop/portal/desktop` |
| Interface | `org.freedesktop.portal.FileChooser` |
| Methods | `OpenFile(s parent_window, s title, a{sv} options) -> o handle` |
| | `SaveFile(s parent_window, s title, a{sv} options) -> o handle` |
| | `SaveFiles(...)` ‚Äî not needed |

The returned object path `o` is a **Request handle**. The method call returns essentially immediately; the real result arrives later as a `Response` signal:

- Interface: `org.freedesktop.portal.Request`
- Signal: `Response(u response, a{sv} results)`
- Emitted on: the Request object path returned by `OpenFile` / `SaveFile`.

After emitting `Response`, the portal unexports the Request object ‚Äî do not try to call `Close` afterwards.

### 3. The `handle_token` race

If you call `OpenFile` and *then* subscribe to the signal on the returned handle, the portal may have already fired `Response` ‚Äî you miss it and hang. The protocol provides `handle_token` so the client can **pre-compute** the Request path and subscribe *before* making the call.

Expected Request path:

```
/org/freedesktop/portal/desktop/request/<SENDER>/<TOKEN>
```

where `<SENDER>` is your unique bus name with the leading `:` stripped and every `.` replaced by `_`. Example: unique name `:1.234` becomes `1_234`.

```java
String uniqueName = conn.getConnection().getUniqueName();    // ":1.234"
String sender     = uniqueName.substring(1).replace('.', '_');
String token      = "jsignpdf_" + ThreadLocalRandom.current().nextInt(1, Integer.MAX_VALUE);
String expected   = "/org/freedesktop/portal/desktop/request/" + sender + "/" + token;
// 1) add signal handler on `expected`
// 2) call OpenFile/SaveFile with options["handle_token"] = token
```

Token must be a valid object-path segment (`[A-Za-z0-9_]+`).

### 4. Options dictionary `a{sv}`

| Key | Sig | Applies to | Meaning |
|---|---|---|---|
| `handle_token` | `s` | both | Pre-computed token. |
| `accept_label` | `s` | both | Confirm button text. |
| `modal` | `b` | both | Request modality vs parent. |
| `multiple` | `b` | OpenFile | Allow multi-select. |
| `directory` | `b` | OpenFile | Pick a folder. Not used by JSignPdf. |
| `current_name` | `s` | SaveFile | Suggested filename. |
| `current_folder` | `ay` | SaveFile | Starting dir, **NUL-terminated byte array**. |
| `current_file` | `ay` | SaveFile | Existing file to overwrite, **NUL-terminated byte array**. |
| `filters` | `a(sa(us))` | both | See below. |
| `current_filter` | `(sa(us))` | both | Preselected filter. |

Filter type mapping:

| `type` (uint32) | `pattern` meaning | Example |
|---|---|---|
| `0` | Glob | `*.pdf` |
| `1` | MIME type | `application/pdf` |

JavaFX's `ExtensionFilter("PDF Files", "*.pdf")` maps directly with `type=0`. Multi-glob filters ("Image Files" ‚Üí png, jpg, jpeg, gif) emit one label with multiple `(0, pattern)` tuples. Backends may regroup filters visually ‚Äî this is cosmetic.

Byte-array paths (`ay`): UTF-8 bytes of the absolute path + trailing `0x00`. ‚öÝ verify behaviour on non-UTF-8 locales.

### 5. Response dictionary

The `Response` signal carries `(uint32 response, a{sv} results)`.

| `response` | Meaning | Action |
|---|---|---|
| `0` | Success | Read `results["uris"]`. |
| `1` | User cancelled | Return null. `results` empty. |
| `2` | Error | Log and fall back. |

Result keys we care about: `uris as` ‚Äî array of `file://` URIs (always present on success; length 1 for single-select and Save). `choices` and `current_filter` in the response are ignored.

### 6. URI and Document-portal path handling

Portal returns percent-encoded `file://` URIs:

```java
Path p = Path.of(URI.create(uri));
```

Inside Flatpak, files outside the sandbox's static filesystems appear via the **Document portal** at `/run/user/<uid>/doc/<docid>/<original-name>`. That path is a real FUSE mount ‚Äî `Files.newInputStream`, `new FileOutputStream` all work. **Do not un-map to the host path**; do not `toRealPath()`. JSignPdf can read/write it as-is.

### 7. Parent window handle

The portal's `parent_window` argument wants `x11:0x<hex-xid>` or `wayland:<handle>` (from `xdg_foreign`). JavaFX has **no public API** for either. Reflection into `com.sun.glass.ui.Window#getNativeWindow()` works on X11 but requires `--add-opens` and is useless on Wayland.

**v1:** pass `""` (no parent). Dialog will open non-modal relative to the Java window. Deferred to v2 with a TODO in code.

### 8. dbus-java

`com.github.hypfvieh:dbus-java-core` 5.1.x requires Java 17+ (fine on 21).

```xml
<dependency>
  <groupId>com.github.hypfvieh</groupId>
  <artifactId>dbus-java-core</artifactId>
  <version>${dbus-java.version}</version>
</dependency>
<dependency>
  <groupId>com.github.hypfvieh</groupId>
  <artifactId>dbus-java-transport-native-unixsocket</artifactId>
  <version>${dbus-java.version}</version>
</dependency>
```

Use the **native-unixsocket** transport (pure-Java, JEP 380). Avoid `dbus-java-transport-jnr-unixsocket` ‚Äî drags in JFFI native libs, complicating the Flatpak manifest.

```java
DBusConnection conn = DBusConnectionBuilder.forSessionBus().build();
// ‚Ä¶ register signal handler, call method ‚Ä¶
conn.disconnect(); // on shutdown
```

In Flatpak, `$DBUS_SESSION_BUS_ADDRESS` is already set by `xdg-dbus-proxy`; no manifest addition beyond standard finish-args.

### 9. Failure modes

| Condition | Detection | Action |
|---|---|---|
| Session bus unreachable | `DBusConnectionBuilder` throws | Cache unreachable, fall back. |
| Portal service not owned | `NameHasOwner` returns false | Cache unreachable, fall back. Probe once at first use. |
| User cancel | `response == 1` | Return null (same as JavaFX). |
| Slow backend / no Response | See ¬ßImplementation ¬ß4 (timeouts) | Cancel + fall back. |
| `current_folder` / `current_file` unreachable in sandbox | Portal silently ignores | Do not validate beforehand. |

### 10. Non-goals at the portal level

- No in-dialog progress, no custom widgets, no preview thumbnail control.
- Directory picking (`directory=true`) not used.
- `SaveFiles` not used (no multi-save caller).

|---|---|---|---|---|---|---|---|
| 1 | `MainWindowController.onFileOpen()` | `jfx.gui.dialog.openPdf.title` | Open | `PDF Files ‚Üí *.pdf` | ‚Äî | `stage` | Input PDF. |
| 2 | `MainWindowController.onSaveAs()` | `jfx.gui.dialog.selectOutputPdf` | Save | `PDF Files ‚Üí *.pdf` | derived from `signingVM.outFile` / `suggestedOutFileFor()`; sets both initial dir and name | `stage` | **Only site using initialDir + initialName.** |

| 5 | `SignaturePropertiesController.onBrowseOutFile()` | `jfx.gui.dialog.selectOutputPdf` | Save | `PDF Files ‚Üí *.pdf` | ‚Äî | scene window | Duplicates #2's title key, but no initial-name logic. |

No site uses `showOpenMultipleDialog`. All expect a single `File` or `null`.

### Invariants the wrapper MUST preserve

1. **Localized title passed verbatim** (from `ResourceProvider`).
2. **Filter order and labels preserved exactly.** Site 3's "All Files"-first ordering is intentional ‚Äî must not be reshuffled, alphabetized, or deduplicated.
3. **Default filter mapping.** JavaFX's default is the first-added filter (visually selected), but `getSelectedExtensionFilter()` returns null unless set explicitly. Only set portal `current_filter` when the caller called `setSelectedFilter`; otherwise omit the key and let the portal show the first filter naturally.
4. **Owner `Window` is mandatory input.** Wrapper signature enforces it. Parent handle is `""` in v1 (see Linux ¬ß7).
5. **Return types match JavaFX exactly** ‚Äî `File` / `List<File>` / `null`. Multi-select API exists for symmetry even though no caller uses it.
6. **`setInitialDirectory` and `setInitialFileName` preserved** ‚Äî currently only Site 2 uses them.
7. **Save dialog auto-appends the default extension** when the user types a bare filename ‚Äî JavaFX does, portal does not, wrapper MUST.
8. **Blocking contract on the FX thread preserved.** Call sites use the return value synchronously; the wrapper mustn't force them to async.

### Out-of-scope UX gaps (acknowledged, not fixed here)

- Sites 1, 3, 4, 5, 6, 7 have no `initialDirectory`. A "last-used directory" default would be a real UX win ‚Äî separate change.
- Sites 6 and 7 have no filters. Adding filters is a separate UX task.
- Wayland modality degradation due to `parent_window=""` ‚Äî known cost, revisit in v2.

### Three UX risks the implementation must address

1. **Silent portal failure ‚Üí broken-looking dialog.** Log, show a one-time notice, disable portal for the session.
2. **Double-dialog risk.** Controllers MUST call the wrapper and MUST NOT instantiate `javafx.stage.FileChooser` directly after migration.
3. **Late portal response after owner window closed.** The async `Response` signal may arrive after the user has closed the stage. Result must be discarded cleanly ‚Äî no `NullPointerException`, no late `setText` on a detached scene.

## Implementation Design

### 1. Abstraction: `NativeFileChooser` facade

New type `net.sf.jsignpdf.fx.util.NativeFileChooser` ‚Äî not an extension of `javafx.stage.FileChooser`, which is `final` and exposes no backend seam. A facade also lets us own the extension-filter representation (the portal backend needs raw glob strings; JavaFX's internal normalization is not useful).

API is a near drop-in so migration diffs are mechanical:

```java
public final class NativeFileChooser {
    public NativeFileChooser setTitle(String t);
    public NativeFileChooser addFilter(ExtensionFilter f);
    public NativeFileChooser setSelectedFilter(ExtensionFilter f);
    public NativeFileChooser setInitialDirectory(File d);
    public NativeFileChooser setInitialFileName(String n);

    public File        showOpenDialog(Window owner);
    public List<File>  showOpenMultipleDialog(Window owner);
    public File        showSaveDialog(Window owner);

    public static record ExtensionFilter(String description, List<String> extensions) {
        public static ExtensionFilter of(String desc, String... exts) { ‚Ä¶ }
    }
}
```

Return types mirror JavaFX exactly (`File` / `List<File>` / `null` on cancel).

**Before / after ‚Äî `MainWindowController.onFileOpen()`:**

```java
// Before
FileChooser fc = new FileChooser();
fc.setTitle(i18n("jfx.gui.dialog.openPdf.title"));
fc.getExtensionFilters().add(new FileChooser.ExtensionFilter("PDF Files", "*.pdf"));
File f = fc.showOpenDialog(stage);

// After
File f = new NativeFileChooser()
    .setTitle(i18n("jfx.gui.dialog.openPdf.title"))
    .addFilter(ExtensionFilter.of("PDF Files", "*.pdf"))
    .showOpenDialog(stage);
```

**`onSaveAs()`** (only site with initial dir + name):

```java
File f = new NativeFileChooser()
    .setTitle(i18n("jfx.gui.dialog.selectOutputPdf"))
    .addFilter(ExtensionFilter.of("PDF Files", "*.pdf"))
    .setInitialDirectory(currentOutputDir())
    .setInitialFileName(suggestedOutputName())
    .showSaveDialog(stage);
```

### 2. Runtime dispatch

Per-call backend choice, made inside the facade:

```
backend = override == FX     ? FX
        : override == PORTAL ? PORTAL
        : (isLinux && Sandbox.isSandboxed()
                   && portalReachable()
                   && !portalDisabledThisSession) ? PORTAL
        : FX
```

- `Sandbox.isSandboxed()` lives in `net.sf.jsignpdf.fx.util.Sandbox` (new utility, implements the Linux ¬ß1 check).
- `portalReachable()` does a `NameHasOwner("org.freedesktop.portal.Desktop")` probe on first use only; caches a volatile boolean for the session. If DBus setup itself throws, cache `false` and log at **DEBUG** (not INFO ‚Äî this path is hit by every non-sandboxed Linux user).
- `portalDisabledThisSession` is a volatile boolean, set when a portal call fails unrecoverably.
- Developer override `-Djsignpdf.filechooser=fx|portal|auto` (default `auto`). Documented in the Flatpak/Snap troubleshooting section only ‚Äî not in main user docs; it's a pressure valve, not a feature. On non-Linux, a `portal` setting **silently degrades to FX** and logs at DEBUG. Never throw on a debug flag.

### 3. Portal backend: `PortalFileChooserBackend`

New package `net.sf.jsignpdf.fx.portal`. A lazy DBus session-connection singleton held for the app's lifetime, disconnected via a **JVM shutdown hook** ‚Äî `Application.stop()` is not guaranteed on `kill`, crashes, or `System.exit` from non-FX threads; a shutdown hook is the belt.

```java
final class PortalFileChooserBackend {
    private static volatile DBusConnection CONN;
    private static DBusConnection conn() throws DBusException {
        DBusConnection c = CONN;
        if (c == null) synchronized (PortalFileChooserBackend.class) {
            if ((c = CONN) == null) {
                c = DBusConnectionBuilder.forSessionBus().build();
                Runtime.getRuntime().addShutdownHook(new Thread(() -> {
                    try { c.disconnect(); } catch (Exception ignored) {}
                }, "jsignpdf-dbus-shutdown"));
                CONN = c;
            }
        }
        return c;
    }
    Optional<List<Path>> open(Request req) throws Exception;
    Optional<List<Path>> save(Request req) throws Exception;
}
```

Per-call flow:

1. Compute `handle_token` and the expected Request path (Linux ¬ß3).
2. Register the `Request.Response` signal handler on that path **before** the method call. Keep a per-call `AtomicBoolean settled` ‚Äî signal handler no-ops if already set (blocks late-arrival races with the fallback path).
3. Build options `a{sv}`: `handle_token`, `modal=true`, `multiple` (open-multi only), `filters`, `current_filter` **only if caller set one**, plus `current_folder` / `current_name` / `current_file` (NUL-terminated UTF-8 `ay`) for Save.
4. Invoke `OpenFile` / `SaveFile` through a dbus-java proxy interface:

```java
@DBusInterfaceName("org.freedesktop.portal.FileChooser")
interface FileChooser extends DBusInterface {
    DBusPath OpenFile(String parentWindow, String title, Map<String, Variant<?>> options);
    DBusPath SaveFile(String parentWindow, String title, Map<String, Variant<?>> options);
}
```

5. Await the `Response` signal. **Timeout discipline:** keep a short **10 s timeout on the initial method call** (handle acquisition ‚Äî a flaky bus manifests here). Place **no timeout on the Response wait** ‚Äî users legitimately spend minutes browsing. A user who walks away isn't a failure; the dialog will eventually `response=1` or the app will exit.
6. On `response=0`, parse `results["uris"]` and `Path.of(URI.create(uri))`. On `response=1`, return `null`. On `response=2`, treat as unrecoverable and fall back.

**Filter marshalling.** A small `FilterMarshaller` in the same package maps each `ExtensionFilter` to a `(label, [(0, pattern)‚Ä¶])` tuple, preserving insertion order (pinning "All Files" to index 0 in Site 3 is the contract). Unit-testable: `FilterMarshaller` produces plain Java records that the dbus-java binding wraps; no DBus dependency in the test.

### 4. Threading and the blocking contract

All current call sites invoke `show*Dialog` synchronously on the JavaFX Application Thread and use the return value inline. That contract is preserved.

Rejected alternatives:

- **`Platform.enterNestedEventLoop`** ‚Äî reentrancy inside controller handlers, focus-restoration bugs, the difficulty of reasoning about pending `runLater` tasks during a nested pump. Bad fit for a modal dialog that doesn't need any FX interaction while open.
- **Async `CompletableFuture<File>` API** ‚Äî rippling into every caller for marginal benefit; the dialog is modal anyway.

Adopted:

```java
CompletableFuture<Optional<List<Path>>> fut = backend.saveAsync(req);
Optional<List<Path>> result;
try {
    result = fut.get();                    // no deadline
} catch (ExecutionException ee) {          // DBus-level failure
    markPortalDisabled();
    return fallbackToFx(req);
}
```

The dbus-java worker drives the DBus socket and dispatches the signal; on signal, the future completes; the FX thread unblocks. This blocks the FX thread during the dialog ‚Äî exactly what JavaFX's own `FileChooser.showOpenDialog` already does.

**Late-response guard (two layers):**

- Before returning a result, check `owner != null && !owner.isShowing()` ‚Üí return `null`.
- The `AtomicBoolean settled` per call (¬ß3) ensures the signal handler no-ops if the facade already moved to the FX fallback path.

No spinner during the blocking wait. JavaFX's chooser has the same latency on slow backends and there's no precedent to introduce one.

### 5. Save-dialog extension auto-append

Portal does not auto-append the default extension like JavaFX does. Reproduce in `NativeFileChooser.showSaveDialog` after the backend returns:

- If the filename already contains a `.` after the last `/` ‚Üí unchanged (honors explicit user intent, including an extension that doesn't match the filter).
- Else inspect `selectedFilter` (or `filters.get(0)`). If it has exactly one glob matching `*.<ext>` ‚Üí append `.<ext>`.
- Else (multi-glob filter like "Image Files", or no filter) ‚Üí unchanged. Guessing is worse than doing nothing.

Pure string logic; unit-tested with no running portal.

### 6. Error handling and fallback policy

| Condition | Action |
|---|---|
| Build-time: dbus-java missing | Compile error. |
| Non-sandboxed run | Portal code never reached. |
| Portal unreachable at startup | Cache `portalReachable=false`. Session runs on FX. One **DEBUG** log line (not INFO ‚Äî spams every non-sandboxed Linux user). |
| Portal call fails (DBus error, `response=2`, signal never arrives within reasonable time) | Log `WARNING` tagged with a UUID. **Immediately** open the FX fallback chooser so the user's flow isn't blocked. When the fallback closes, **then** show a one-time non-modal `Alert(WARNING)` explaining what happened. Set `portalDisabledThisSession=true`. |
| User cancel (`response=1`) | Return `null`. No log, no Alert. |
| Signal after owner closed | Dropped by the `settled` flag and/or the `isShowing` guard. |

**One-strike-per-session, no settings-level retry.** A "Retry native dialogs" preference would add UI surface for an edge case; a restart is the reset. Vetoed.

**Fallback Alert copy.** Avoid "portal", "native", "fallback". Use:

> **Title:** File dialog
> **Body:** Could not open the system file dialog. Using the built-in one instead.

New i18n resource keys: `jfx.gui.dialog.portalFallback.title`, `jfx.gui.dialog.portalFallback.text`.

### 10. Phased rollout

Each phase is independently releasable and revertible.

1. **Phase 1 ‚Äî facade only.** Introduce `NativeFileChooser`, backed exclusively by `javafx.stage.FileChooser`. Migrate all 7 call sites. Verify no regression on any OS.
2. **Phase 2 ‚Äî portal backend.** Add `Sandbox`, `PortalFileChooserBackend`, dispatch logic, dbus-java dependencies. Manual Flatpak testing on GNOME + KDE.
3. **Phase 3 ‚Äî Flatpak hardening.** Drop `--filesystem=home` from the manifest, reply on the Flathub PR.

## UX Review & Design Discipline

This section contains the final UX review and closes the open questions.

### Invariant compliance check

1. **Localized title** ‚Äî honored.
2. **Filter order & labels** ‚Äî honored *with the caveat* that marshalling must iterate filters in insertion order. A unit test pinning Site 3's "All Files" at index 0 post-marshal is required.
3. **Default filter ‚Üí `current_filter`** ‚Äî **revised from the initial proposal.** `filters.get(0)` is wrong as an implicit default: JavaFX's `getSelectedExtensionFilter()` returns `null` unless set explicitly. Only populate portal `current_filter` when the caller called `setSelectedFilter`; otherwise omit the key.
4. **Owner `Window`** ‚Äî honored; the signature enforces it.
5. **Return types** ‚Äî honored.
6. **Initial dir + name** ‚Äî honored.
7. **Save auto-append** ‚Äî honored. The "exactly one `*.ext` glob" rule handles `*.pdf` correctly. `*.tar.gz` is a non-issue (no such filter exists). Explicit extensions that don't match the filter are preserved, honoring user intent.
8. **Blocking contract** ‚Äî honored; rejecting `enterNestedEventLoop` is the right call.

**Late-response guard was insufficient as first proposed.** Owner-visibility doesn't cover the race where the FX fallback has already opened (owner still showing). Resolved by the per-call `AtomicBoolean settled` flag ‚Äî implementation section updated.

### Risks

**Silent-failure sequencing** was wrong as first written: Alert-then-FX-chooser = two interruptions for one action. Resolved by opening the FX fallback immediately and deferring the Alert until AFTER the fallback closes. Shown only on the first failure of the session.

**Double-dialog** is covered by the `settled` flag.

**Late-response** is covered by the `settled` flag plus the `isShowing` check.

### Open questions ‚Äî resolved

| # | Question | Resolution | Rationale |
|---|---|---|---|
| 1 | Document `-Djsignpdf.filechooser` for users? | Mention in Flatpak/Snap troubleshooting only | Pressure valve, not a feature. |
| 2 | Fallback notice: `Alert(WARNING)` vs toast vs banner? | `Alert(WARNING)`, once per session, **deferred** until after the fallback chooser closes | Toast infrastructure doesn't exist and isn't worth building for this. |
| 3 | DBus teardown: shutdown hook vs `Application.stop()`? | JVM shutdown hook | `Application.stop()` isn't guaranteed on kill / crash exits. |
| 4 | `portalDisabledThisSession` reset via settings action? | **No.** One-strike-per-session; restart is the reset | UI surface for an edge case. |
| 5 | xdg-foreign parent-window handle in v1? | v2 | Parent=`""` is acceptable; modal-to-nothing is a known portal wart. |

### Wording & polish

- Fallback Alert: **"Could not open the system file dialog. Using the built-in one instead."** Title **"File dialog"**. New keys `jfx.gui.dialog.portalFallback.title` and `.text`. Avoid "native", "portal", "fallback" ‚Äî jargon.
- Non-Linux with `-Djsignpdf.filechooser=portal`: **silently degrade to FX**, DEBUG log. Never throw on a debug flag.
- Slow portal backends: **no spinner.** JavaFX's own chooser has the same latency. Revisit only if reported.
- Log level for portal-unreachable at startup: **DEBUG**, not INFO ‚Äî INFO would spam every non-sandboxed Linux dev.
- Timeout discipline: **no timeout** on the Response wait (users legitimately browse for minutes). Keep a **10 s timeout** only on the initial method call (handle acquisition).

### Scope discipline

- No `showDirectoryDialog`.
- No settings-level portal retry.
- xdg-foreign is v2.
- No "last-used directory" defaulting as part of this change.
- Seven sites, one facade. Done.

## Summary of key decisions

| Decision | Choice |
|---|---|
| Activation | Linux AND inside Flatpak/Snap sandbox only |
| Detection | `/.flatpak-info` ‚à® `FLATPAK_ID` ‚à® `SNAP` |
| Transport | `dbus-java-core` 5.1.x + `dbus-java-transport-native-unixsocket` |
| API | New `NativeFileChooser` facade (not a `FileChooser` subclass) |
| Dispatch | Per-call, with session-level `portalReachable` + `portalDisabled` caches |
| Threading | Blocking on FX thread (`future.get()` with no deadline) |
| Timeouts | 10 s on method call; none on Response wait |
| Parent window | `""` in v1; xdg-foreign deferred to v2 |
| Filters | Insertion order preserved; `current_filter` only if caller set one |
| Save auto-append | Implemented in facade for single-glob filters |
| Failure UX | FX fallback first, then deferred one-time Alert |
| Session reset | Restart only, no settings action |
| Rollout | 3 phases: facade ‚Üí portal backend ‚Üí drop `--filesystem=home` |
| Developer override | `-Djsignpdf.filechooser=fx\|portal\|auto`, docs in troubleshooting section only |
