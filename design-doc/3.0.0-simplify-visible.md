# Simplify the visible-signature settings panel (JavaFX UI)

Status: implemented
Target release: 3.0.0
Scope: JavaFX GUI only. Swing GUI, CLI, and `BasicSignerOptions` semantics unchanged.

## 1. Context and motivation

The current JavaFX **Signature settings** panel exposes every visible-signature knob JSignPdf has ever supported: render mode, L2 text, L4 text, font size, signature image, background image, and Acro6 layers. Most of these are power-user settings. Newcomers see a dense form where almost all fields are optional, and getting a good-looking signature requires trial-and-error across three or four of them.

In practice the overwhelmingly common case is: *"put some text on top of a logo."* That is exactly what `RenderMode.DESCRIPTION_ONLY` does when combined with a background image. The other render modes (`GRAPHIC_AND_DESCRIPTION`, `SIGNAME_AND_DESCRIPTION`) and the `imgPath` field duplicate what users already get by setting `bgImgPath`.

This design trims the JavaFX panel to the two fields that matter and adds a preview for the background image so users can see what they picked before signing.

## 2. Design goals and non-goals

**Goals**

- Expose only two signature-appearance fields in the JavaFX UI: the signature text (multiline) and the background-image path.
- Show a live preview of the selected background image next to / below the path field.
- Keep `BasicSignerOptions` and the signing pipeline unchanged — this is a UI simplification, not a feature removal.
- Make the simplified UI produce deterministic, canonical option values (see §3).

**Non-goals**

- No changes to the Swing GUI or the CLI.
- No changes to `RenderMode`, `BasicSignerOptions`, or the PDF rendering code.
- No change to signature placement. Placement is already owned by `SignaturePlacementViewModel` (relative rectangle), the page selector in the main toolbar, and the drag-to-place interaction on the PDF preview. None of that is in this panel and none of it moves.
- No change to the sibling **Signature properties** accordion section (`SignaturePropertiesController` / `SignatureProperties.fxml`). That section holds signature *metadata* (output file, hash algorithm, cert level, signer name, reason, location, contact) and is independent of the appearance simplification here.
- No new rendering features (e.g. background opacity, text alignment). Out of scope.

## 3. Canonical option values

When a visible signature is enabled through the simplified UI, the view-model writes these fixed values into `BasicSignerOptions` on every `syncToOptions`:

| Field | Value |
|---|---|
| `renderMode` | `RenderMode.DESCRIPTION_ONLY` |
| `l2Text` | user input (may contain `\n`) |
| `bgImgPath` | user input (may be null / empty) |
| `imgPath` | `null` |
| `l4Text` | `null` |
| `l2TextFontSize` | `Constants.DEFVAL_L2_FONT_SIZE` |
| `acro6Layers` | `Constants.DEFVAL_ACRO6LAYERS` |
| `bgImgScale` | `Constants.DEFVAL_BG_SCALE` (already the case) |

This makes the simplified UI a pure subset of the CLI capability. If a user wants non-default values for the hidden fields, they can still set them via CLI flags, config file, or properties preset — the signing path honors them. Only the JavaFX UI does not surface the controls.

**Decided:** legacy values loaded from a config file / preset are read into the view-model (so round-tripping doesn't lose data mid-session), but on `syncToOptions` the hidden fields are always reset to the canonical values above. Saving therefore normalizes a preset to the simplified schema.

## 4. UI changes

### 4.1 `SignatureSettings.fxml`

Remove:

- `Label` + `ComboBox fx:id="cmbRenderMode"`
- `Label` + `TextField fx:id="txtL4Text"`
- `Label` + `TextField fx:id="txtFontSize"`
- `Label` + `HBox` containing `txtImgPath` and its browse button
- `CheckBox fx:id="chkAcro6Layers"`

Replace the single-line L2 `TextField` with a `TextArea` (with prompt text — see §4.5):

```xml
<Label text="%jfx.gui.sig.l2Text"/>
<TextArea fx:id="txtL2Text" wrapText="true" prefRowCount="4"
          promptText="%jfx.gui.sig.l2Text.prompt"/>
```

Add a preview underneath the background-image row:

```xml
<Label text="%jfx.gui.sig.backgroundImage"/>
<HBox spacing="4">
    <TextField fx:id="txtBgImgPath" HBox.hgrow="ALWAYS"/>
    <Button onAction="#onBrowseBgImage" minWidth="30"> ... </Button>
    <Button onAction="#onClearBgImage" minWidth="30"> ... </Button>  <!-- new -->
</HBox>
<StackPane fx:id="bgImgPreviewPane" minHeight="120" maxHeight="180"
           styleClass="preview-pane">
    <ImageView fx:id="bgImgPreview" preserveRatio="true"
               fitWidth="260" fitHeight="160"/>
    <Label fx:id="bgImgPreviewPlaceholder" text="%jfx.gui.sig.bgImage.none"/>
</StackPane>
```

Preview behavior:

- Empty path → show placeholder label, hide `ImageView`.
- Valid path → load via `new Image(file.toURI().toString(), 260, 160, true, true, true)` (async, preserveRatio, smooth). Show `ImageView`, hide placeholder.
- Load error (`image.errorProperty()`) → show placeholder with "cannot load image" message (new resource key).
- Path field changes are debounced (~150 ms via `Platform.runLater` + a pending-flag, or a simple `ChangeListener` — we reload on each stable value).

### 4.2 `SignatureSettingsController.java`

- Drop fields: `cmbRenderMode`, `txtL4Text`, `txtFontSize`, `txtImgPath`, `chkAcro6Layers`, `onBrowseImage()`.
- Change `txtL2Text` type from `TextField` to `TextArea`.
- Add fields: `bgImgPreview` (`ImageView`), `bgImgPreviewPlaceholder` (`Label`), `bgImgPreviewPane` (`StackPane`).
- Add `onClearBgImage()` that sets `txtBgImgPath.setText("")`.
- Replace the render-mode combobox population and font-size listener with a single listener on `txtBgImgPath.textProperty()` that drives the preview.
- **Enable/disable, don't show/hide.** Replace the existing visibility toggle:
  ```java
  // old
  visibleSigPane.managedProperty().bind(visibleSigPane.visibleProperty());
  chkVisibleSig.selectedProperty().addListener((obs, o, n) -> visibleSigPane.setVisible(n));
  visibleSigPane.setVisible(false);
  // new
  visibleSigPane.disableProperty().bind(chkVisibleSig.selectedProperty().not());
  ```
  The signature text area and background-image preview stay laid out and readable when visible-signature is off. The image preview keeps rendering even while the pane is disabled — JavaFX bindings run regardless of `disabled`.

Binding changes in `bindToViewModel`:

- Keep: `chkVisibleSig ↔ visibleProperty`, `txtL2Text ↔ l2TextProperty`, `txtBgImgPath ↔ bgImgPathProperty`.
- Drop bindings for: `renderMode`, `l4Text`, `l2TextFontSize`, `imgPath`, `acro6Layers`.

### 4.3 `SigningOptionsViewModel.java`

- Keep all existing visible-signature properties (`renderModeProperty`, `l4TextProperty`, `l2TextFontSizeProperty`, `imgPathProperty`, `acro6LayersProperty`) — `syncFromOptions` still reads them so nothing is lost mid-session, and no other consumer of the view-model needs to change.
- In `syncToOptions`, hard-code the canonical values instead of forwarding the property values:
  ```java
  opts.setRenderMode(RenderMode.DESCRIPTION_ONLY);
  opts.setImgPath(null);
  opts.setL4Text(null);
  opts.setL2TextFontSize(Constants.DEFVAL_L2_FONT_SIZE);
  opts.setAcro6Layers(Constants.DEFVAL_ACRO6LAYERS);
  ```
- `resetToDefaults` already sets these to their canonical values — no change there.

### 4.5 "Default signature description" — handling empty `l2Text`

We want to distinguish two user intents that today both map to empty input:

1. **"I don't care, give me the default"** — standard JSignPdf behavior: generate a description from signer name, date, reason, location.
2. **"I want a blank signature rectangle"** — no auto-generated text, just the background image (and nothing else).

The signing pipeline already makes this distinction (`SignerLogic.java:307` — `StringUtils.isEmpty(options.getL2Text())` → default is applied). We surface it in the UI by convention, without adding a checkbox:

| TextArea content | `l2Text` written to options | Result in PDF |
|---|---|---|
| empty | `null` | default description is generated |
| `" "` (single space) | `" "` | literal space — effectively blank visible text |
| any non-empty text | that text verbatim | rendered verbatim |

**Communicating the convention to the user** — `TextArea.promptText` (greyed-out text shown only while the area is empty, not part of `textProperty()`) + a tooltip on the `TextArea`:

- promptText: `Default signature description`
- tooltip: `Leave empty for the default description. Type a space (or any text) to override.`

Rationale for picking the prompt-text approach over Swing's explicit "use default" checkbox:

- One fewer control on a panel whose whole point is to be simpler than Swing's.
- `promptText` is JavaFX's canonical placeholder mechanism — exactly what it's for.
- The backend already collapses null and `""` to "use default", so no code change in the signing path.

Rationale for the "space = blank" convention:

- Everything else is already wired up; this falls out of the existing pipeline for free.
- The tooltip makes it discoverable for the small minority who actually want a blank visible-text signature.

## 5. Multiline signature text

Multiline `l2Text` already works end-to-end — the Swing UI has been shipping it for years. `VisibleSignatureDialog` uses a `JTextArea` (`taL2Text`, 5 rows; see `VisibleSignatureDialog.java:347,752-753`), writes the raw multiline string into `options.setL2Text(...)`, and `SignerLogic` hands it verbatim to OpenPDF's `PdfSignatureAppearance#setLayer2Text(...)`, which lays out embedded newlines itself.

The JavaFX `TextField` was the only thing holding this back. Swapping it to a `TextArea` closes the gap — no render-path changes, no follow-up issue needed.

## 6. Persistence / backwards compatibility

- Existing `~/.JSignPdf` and preset files may contain non-default values for `renderMode`, `imgPath`, `l4Text`, `fontSize`, `acro6Layers`. Those are loaded into the view-model as before (so presets remain loss-free on *read*).
- The next time the user saves, the hidden fields are normalized to canonical values (see §4.3). This is intentional: the simplified UI owns the file now.
- Users who need the hidden features continue to use the CLI or edit the properties file by hand.
- No migration script needed.

## 7. Testing plan

- Unit tests for `SigningOptionsViewModel.syncToOptions` / `syncFromOptions` covering the canonical-value overwrite (Option A).
- Manual UI pass:
  - toggle visible signature on/off; sub-panel stays visible but becomes disabled/enabled, preview still renders while disabled.
  - leave the text area empty, sign → auto-generated default description appears in PDF.
  - type a single space in the text area, sign → no auto-generated text appears; rectangle shows only the background image.
  - paste a multiline string into the text area; sign a PDF; verify both lines render.
  - set, change, and clear the background image; preview updates each time.
  - point path at a missing file → error placeholder appears.
  - point path at a non-image file (e.g. a PDF) → error placeholder appears.
  - load a legacy preset with `renderMode=GRAPHIC_AND_DESCRIPTION`, confirm UI still works; save, reload, verify render mode is now `DESCRIPTION_ONLY` on disk.
